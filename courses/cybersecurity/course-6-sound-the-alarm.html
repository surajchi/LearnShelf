Events vs. Incidents & the NIST Lifecycle

📖DefinitionDEF-501

Incident (NIST definition): An occurrence that actually or imminently jeopardizes, without lawful authority, the confidentiality, integrity, or availability of information or an information system, or that constitutes a violation or imminent threat of violation of security policies, procedures, or acceptable use policies.

Event: An observable occurrence in a system or network — not inherently malicious.

Explain Like I'm Five — The Password Reset

A password reset is an event — it happens all the time, and it's completely normal. But if someone requests a hundred password resets for accounts they don't own, in the middle of the night, from an unfamiliar country — that pattern of events becomes a security incident, because now it threatens the confidentiality of those accounts.

❗ImportantIMP-501

This lifecycle is cyclical, allowing overlapping steps as new information emerges — it is not a strict, one-directional checklist. A team might return to Detection and Analysis mid-containment if new evidence surfaces. It also directly implements the last three of the NIST CSF's five core functions (Course V): Detect, Respond, Recover.

📖DefinitionDEF-502

Incident Handler's Journal: A tool for organized documentation of incident details, recording vital information during response and investigation — one of the first tools in a security analyst's toolbox.

An incident investigation aims to determine the same five questions a detective would ask: who triggered the incident, what happened, when it took place, where it took place, and why it occurred.

Applying the Lifecycle: A Vishing Scenario

🛰️SOC Example — Vishing Response WalkthroughSOC-501

Preparation — The organization sets up uniform email conventions, creates a collaborative environment where employees feel comfortable asking questions, and provides quarterly cybersecurity training.

Detection & Analysis — Security professionals filter external emails to flag attachments like voicemails, and reference an incident response plan when a suspicious voice message arrives.

Containment, Eradication & Recovery — The team communicates with the alleged sender to confirm origin, and provides employees an easy way to report and contain suspicious messages.

Post-Incident Activity — The playbook gets updated to highlight new red flags, and permission oversight processes are reviewed and adjusted.

Incident Response Teams: CSIRT & SOC

📖DefinitionDEF-503

Computer Security Incident Response Team (CSIRT): A specialized group of security professionals trained in incident management and response — sometimes called an Incident Handling Team (IHT) or Security Incident Response Team (SIRT) depending on the organization.

Command, Control, Communication: Command is having appropriate leadership overseeing the response. Control is the ability to manage technical aspects — coordinating resources, assigning tasks. Communication is keeping stakeholders informed.

Three Core CSIRT Roles
RoleResponsibilities
Security AnalystContinuously monitors the environment; analyzes/triages alerts; performs root-cause investigations; escalates or resolves
Technical LeadManages all technical aspects of response; determines root cause; creates/implements containment-eradication-recovery strategy; aligns priorities with business needs
Incident CoordinatorCoordinates with external departments (legal, PR, HR); keeps lines of communication open across all involved teams
📖DefinitionDEF-504

Security Operations Center (SOC): An organizational unit dedicated to monitoring networks, systems, and devices for security threats — often called "the blue team" for its defensive role. May exist as its own unit or within a CSIRT.

SOC Tiers & Specialized Roles
RoleResponsibilities
Tier 1 (L1) AnalystMonitors, reviews, and prioritizes alerts by severity; creates/closes tickets; escalates to Tier 2/3
Tier 2 (L2) AnalystReceives escalated tickets; conducts deeper investigations; configures/refines security tools; reports to SOC Lead
Tier 3 (L3) SOC LeadManages team operations; performs advanced detection (malware/forensics analysis); reports to SOC Manager
SOC ManagerHires/trains/evaluates the team; sets performance metrics; develops incident/compliance/audit reports; communicates to executives
Forensic InvestigatorsUsually L2/L3; collect, preserve, and analyze digital evidence to determine what happened
Threat HuntersTypically L3; proactively detect, analyze, and defend against new/advanced threats using threat intelligence

Incident Response Plan: A document outlining the procedures to take in each step of incident response, built from three basic elements — policies, standards, and procedures (recall Course V, Module 1).

Incident response plans are not static. Regular review and testing — tabletop exercises, simulations — help familiarize teams with the plan and identify gaps. Some organizations are required to conduct specific exercises for regulatory compliance, not just internal readiness.

IDS, IPS & EDR Compared

📖DefinitionDEF-506

Endpoint Detection and Response (EDR): An application installed on endpoints that monitors, records, and analyzes activity to identify, alert, and respond to suspicious behavior — using machine learning/AI-powered behavioral analysis and automation to stop attacks without manual intervention.

Four Detection Categories
CategoryMeaning
True positiveAn alert correctly detecting a real attack
True negativeNo malicious activity exists, and no alert is triggered — correct
False positiveAn alert incorrectly flags something as malicious that isn't — wastes analyst time/resources
False negativeMalicious activity happens but goes undetected — dangerous, since the team remains unaware and vulnerable
🎯Exam TipEXM-501

Between the two errors, false negatives are generally considered more dangerous than false positives — a false positive costs time; a false negative leaves a real attack completely unaddressed. When tuning detection systems, security teams often accept somewhat more false positives to reduce false negatives.

SIEM & SOAR: Alert and Event Management

Explain Like I'm Five — The Car Dashboard

A SIEM tool is like a car's dashboard — it doesn't show you every single mechanical detail of the engine, but it gives you a high-level, at-a-glance view of speed, fuel, and warning lights so you know exactly when something needs your attention.

The Three-Step SIEM Process
StepWhat Happens
1. Collect and aggregate dataData pulled from IDS/IPS, databases, firewalls, applications into one centralized location
2. Normalize dataRemoves non-essential attributes, creates consistency across log records from different sources
3. Analyze dataCompares data against configured rules; matches are categorized and reported as alerts
📖DefinitionDEF-507

SOAR (Security Orchestration, Automation, and Response): A collection of applications, tools, and workflows using automation to respond to security events and incidents — can also track and manage cases, where multiple related incidents are centrally managed together.

Hands-On Labs & Practical Exercises

Lab 1.1 — Event or Incident?

Classify each as an event or an incident: (1) a user resets their own forgotten password, (2) a hundred failed login attempts against a single admin account within one minute, (3) an employee logs into their email from a new laptop they just purchased.

Answer key: 1 — Event (normal, expected). 2 — Incident (threatens confidentiality/integrity, violates expected patterns). 3 — Event, most likely, though it may warrant a closer look if unusual for that user.

Lab 1.2 — Detection Category Drill

Classify each scenario: (1) an IDS alerts on a legitimate software update mistaken for malware; (2) a sophisticated attacker evades detection entirely and exfiltrates data undetected; (3) an IDS correctly flags an active SQL injection attempt.

Answer key: 1 — False positive. 2 — False negative. 3 — True positive.

Scenario Challenge 1.1 — CSIRT Role Assignment

A critical ransomware incident requires deep technical remediation work and close coordination with the legal and PR departments. Which two CSIRT roles are most central here, and why?

Answer: The Technical Lead (managing containment/eradication/recovery strategy and root-cause determination) and the Incident Coordinator (keeping legal, PR, and other departments informed and aligned) — both roles are essential for a critical incident of this scope.

Interview Preparation

🎤Beginner QuestionINT-501

Q: Walk me through the four phases of the NIST incident response lifecycle.

Ideal answer: "Preparation, Detection and Analysis, Containment/Eradication/Recovery, and Post-Incident Activity. It's important to note the lifecycle is cyclical — teams can revisit earlier phases as new information emerges, and lessons learned in Post-Incident Activity feed directly back into Preparation for next time."

🎤SOC Analyst QuestionINT-502

Q: You're a Tier 1 analyst and receive an alert that doesn't clearly fit your playbook. What do you do?

Ideal answer: "I'd first verify whether it's a known false positive pattern by checking history and severity. If it's genuinely ambiguous or high-severity, I'd escalate to a Tier 2 analyst rather than guess — that's exactly what the tiered structure is designed for, and escalating appropriately is a sign of good judgment, not a failure to resolve it myself."

Practice Section

1. Which detection category describes malicious activity that occurs but is never detected?
Correct — a false negative is undetected malicious activity, considered more dangerous than a false positive.
2. True or False: The NIST incident response lifecycle is strictly linear, with no revisiting of earlier phases.
False — the lifecycle is cyclical, allowing overlapping steps as new information emerges.
3. Which SOC role typically performs advanced malware and forensics analysis?
Correct — Tier 3 SOC leads perform advanced detection techniques like malware and forensics analysis.

Revision Notes & Cheat Sheet

⚡30-Second RevisionQR-501

NIST IR lifecycle: Preparation → Detection & Analysis → Containment/Eradication/Recovery → Post-Incident Activity (cyclical). CSIRT roles: security analyst, technical lead, incident coordinator. SOC tiers: L1 (triage) → L2 (deep investigation) → L3 (advanced detection) → manager. Module Summary

This module established the structural backbone of Course VI: the cyclical NIST incident response lifecycle, the teams and roles that carry it out, and the tools (IDS/IPS/EDR, SIEM/SOAR) that make detection and response possible at scale.

Key Takeaways

  • Events are normal occurrences; incidents specifically threaten confidentiality, integrity, or availability.
  • The NIST lifecycle's four phases are cyclical, not linear — overlap and revisiting earlier phases is normal.
  • CSIRTs and SOCs have distinct but complementary structures, each with clearly defined roles.
  • IDS, IPS, and EDR form a capability spectrum: detect-only, detect-and-block, and detect-block-plus-behavioral-analysis.
  • False negatives are generally more dangerous than false positives — undetected threats cause the most damage.

Network Traffic Flows & Baselines

📖DefinitionDEF-508

Network traffic: The amount of data moving across a network, including the type of data transferred (e.g., HTTP).

Network data: The actual data transmitted between devices on a network.

Baseline: A reference point used for comparison, establishing a standard of expected/normal behavior for systems, devices, and networks.

Explain Like I'm Five — The Traffic Jam

Just as commuters learn their city's normal rush-hour patterns and immediately notice when traffic looks wrong for the time of day, security professionals learn a network's normal traffic baseline so abnormal activity — like a sudden 3 a.m. surge — jumps out immediately.

Three Network Components to Monitor
ComponentWhat It Reveals
Flow analysisMovement of packets, protocols, and ports — malicious actors sometimes use uncommon port/protocol pairings for command and control (C2)
Packet payload informationThe actual transmitted data (often encrypted) — reveals possible data exfiltration if sensitive data leaves the network unexpectedly
Temporal patternsTime-based baselines — traffic spikes outside normal business hours are "off baseline" and warrant investigation
📖DefinitionDEF-509

Command and Control (C2): Techniques malicious actors use to maintain communications with compromised systems — for example, using HTTPS over port 8088 instead of its standard port 443 specifically to evade simple port-based monitoring.

❗ImportantIMP-502

A network operations center (NOC) maintains network performance, availability, and uptime — distinct from a SOC, which maintains network security. The two units often work side by side but serve fundamentally different missions; don't conflate them.

Data Exfiltration Attacks

📖DefinitionDEF-510

Data exfiltration: The unauthorized transmission of data from a system — detectable through anomalies like large volumes of outbound traffic leaving a host.

Data Exfiltration — Attacker's Playbook vs. Organization's Defense
Attacker's StepsOrganization's Defense
1. Gain initial access (e.g., phishing)Require MFA to prevent phishing-based access
2. Maintain access via lateral movement (pivoting)Monitor for suspicious behavior — multiple logins from outside IPs
3. Identify valuable assets (PII, IP, financial records)Maintain asset inventory with appropriate security controls
4. Prepare data (compress to bypass controls)Detect unusual data collection via SIEM monitoring
5. Exfiltrate data to chosen destinationBlock IP addresses associated with the attacker via firewall rules
📖DefinitionDEF-511

Lateral movement (pivoting): An attacker's technique for exploring a network and accessing other systems after gaining initial access, in search of valuable assets.

Packets, Headers & Packet Captures, Revisited

Course III introduced packet structure at a conceptual level. This module goes deeper into every field, because packet-level detail is where real investigations actually happen.

📖DefinitionDEF-512

Header: Contains routing information — source/destination IP addresses, packet length, protocol, identification numbers. Packets can have several headers (Ethernet, IP, TCP) depending on protocols used.

Payload: The actual data being delivered — like the content of a letter inside an envelope.

Footer (trailer): Located at the end of a packet, used by Ethernet for error-checking. Most protocols, including IP, do not use footers.

📖DefinitionDEF-513

Packet capture (p-cap): A file containing data packets intercepted from an interface or network — an invaluable artifact during incident investigations, letting analysts build a storyline of events after the fact.

⚠️WarningWRN-501

Using network protocol analyzers to intercept and examine private network communications without permission is illegal in many places. This is a professional and legal boundary, not just an ethical guideline — always operate within documented authorization.

Notice the direct equivalences: IPv4's ToS ≈ IPv6's Traffic Class, and IPv4's TTL ≈ IPv6's Hop Limit. IPv6 simplified and renamed several fields rather than inventing entirely new concepts — recognizing these pairs makes memorizing both headers much easier.

Wireshark Display Filters

📖DefinitionDEF-514

Wireshark: An open-source network protocol analyzer with a graphical user interface, making it easier to visualize network communications than command-line-only tools.

Wireshark Comparison Operators
Operator TypeSymbolAbbreviation

Wireshark Display FilterCopy
ip.addr == 172.21.224.2

📖DefinitionDEF-515

Stream (conversation): The exchange of data between devices using a protocol. Wireshark's "Follow Stream" feature reassembles the transferred data into an easy-to-read format — useful for reviewing the full content of an HTTP request/response exchange.

tcpdump Deep Dive

📖DefinitionDEF-516

tcpdump: A command-line network protocol analyzer, pre-installed on many Linux distributions, used to capture and view network traffic including TCP, IP, and ICMP.

Bash — tcpdump SyntaxCopy
sudo tcpdump [-i interface] [option(s)] [expression(s)]
What this does: sudo runs with elevated permissions (required for packet capture). -i specifies the network interface (-i any captures from all interfaces). Options alter execution; expressions further filter traffic.
Essential tcpdump Options
OptionFunction
-w [file]Write/save captured packets to a p-cap file for later analysis
-r [file]Read a specified packet capture file
-v / -vv / -vvvVerbose mode — increasing levels of detail with each added "v"
-c [number]Count — limits the number of packets captured
-n / -nnDisables name resolution (hostnames / hostnames + ports) — best practice for accuracy and stealth
⚠️WarningWRN-502

Default name resolution uses a reverse DNS lookup — a query looking up the domain name associated with an IP address. If you perform this against an attacker's own system during an investigation, it can alert them that they're being investigated through their own DNS records. Using -n/-nn avoids this risk and is considered best practice.

Bash — tcpdump ExamplesCopy
sudo tcpdump -i any -w packetcapture.pcap
sudo tcpdump -r packetcapture.pcap -v -n
sudo tcpdump -r packetcapture.pcap -n 'ip and port 80'
sudo tcpdump -i any -c 3
What each does: Line 1 captures all traffic and saves to a file. Line 2 reads that file verbosely without name resolution. Line 3 reads the file, filtering for IPv4 traffic on port 80, combined with the and boolean operator. Line 4 captures only the first 3 packets seen on any interface.
Pro tip: Combine short options without a value directly (-vn), but options taking a parameter (-c 3, -r file) can't be combined with others in the same cluster.
Interpreting tcpdump Output — Key Fields
FieldDescription
TimestampHours, minutes, seconds, fractions of a second
Source IP / PortThe packet's origin address and port
Destination IP / PortWhere the packet is being transmitted to
🛡️Blue Team TipBT-501

Boolean operators (and, or, not) combined with parentheses let you construct precise, grouped filter expressions — e.g., ip and (port 80 or port 443) prioritizes evaluating the parenthetical group before applying the IPv4 filter. This grouping logic mirrors what you already know from SQL's WHERE clauses (Course IV).

Interview Preparation

🎤Intermediate QuestionINT-503

Q: Why would you disable name resolution when using tcpdump during an investigation?

Ideal answer: "Two reasons: accuracy and stealth. Name resolution can mislead you — tcpdump might label port 80 traffic as HTTP by default, but it's not always accurate. More importantly, resolving an attacker's IP address performs a reverse DNS lookup, which can alert them that they're under investigation through their own DNS records. Using -n or -nn avoids both problems."

🎤Live Analysis QuestionINT-504

Q: You're given a packet capture and asked to find all DNS traffic from a specific IP address. How would you approach this in Wireshark?

Ideal answer: "I'd combine a protocol filter with an IP filter using the and operator: dns and ip.src == [address]. This narrows the massive capture down to exactly the relevant traffic. I might also follow the stream if I need to see the full DNS query/response exchange in context."

Practice Section

1. Which packet capture format is the default for Unix-like systems and tools like tcpdump?
Correct — Libpcap is the default packet capture library on Unix-like systems.
2. True or False: IPv6's Hop Limit field serves the same purpose as IPv4's Time to Live field.
True — both fields limit how long a packet can travel before being discarded.
3. Which tcpdump flag writes captured packets to a file instead of printing them to the terminal?
Correct — the -w flag writes/saves captured packets to a p-cap file.

Wireshark uses comparison operators plus contains/matches; tcpdump uses -w/-r/-v/-c/-n options plus boolean expressions.

This module gave you the deepest technical skill in the certificate so far: reading and filtering actual network traffic, field by field, using both a GUI tool (Wireshark) and a command-line tool (tcpdump) — the two most common ways a real analyst investigates what's actually happening on the wire.

Key Takeaways

  • Baselines make abnormal traffic visible — flow, payload, and temporal analysis are the three lenses for spotting deviations.
  • Data exfiltration follows a predictable attacker chain, which maps directly onto a matching defensive chain.
  • IPv4 and IPv6 headers share conceptual equivalences despite different field names and counts.
  • Wireshark's display filters and tcpdump's command-line options both accomplish the same core task — precise traffic isolation — through different interfaces.
  • Disabling name resolution (-n/-nn) is a small detail with real investigative and operational-security consequences.

Zero Trust & Indicators of Compromise

📖DefinitionDEF-519

Zero Trust: A security model built on "never trust, always verify" — trust is never automatically granted to any user or device, even inside the network. It combines least privilege, continuous monitoring, and micro-segmentation to reduce the "blast radius" of any breach.

Zero Trust's Five Principles
PrincipleMeaning
No implicit trustEvery user, device, or application is treated as untrusted until proven otherwise
VerificationStrong authentication and security checks required to access resources
Least privilegeMinimum access needed to perform tasks, nothing more
Continuous monitoringOngoing observation to catch unusual behavior quickly
Micro-segmentationSmaller network segments, each with its own access controls, limiting breach impact
📖DefinitionDEF-520

Indicators of Compromise (IoCs): Observable evidence suggesting a potential security incident has already happened — like noticing a valuable was stolen from a car.

Indicators of Attack (IoAs): Observed events indicating a real-time incident in progress, focused on behavioral evidence — an attacker's methods and intentions.

Explain Like I'm Five — Who/What vs. Why/How

IoCs answer who and what after the fact. IoAs answer why and how while it's happening. If a process makes a network connection right now, that's an IoA. The filename of that process and the IP address it contacted are the associated IoCs.

⚠️WarningWRN-503

An indicator of compromise is not automatically confirmation that a security incident occurred — IoCs can also result from human error or system malfunction. Context and further investigation are always required before drawing a conclusion.

The Pyramid of Pain

📖DefinitionDEF-521

Pyramid of Pain (David J. Bianco): A model capturing the relationship between types of IoCs and the level of difficulty attackers face when each type is blocked — the higher up the pyramid, the more "pain" caused to the attacker.

The Seven Levels Explained
LevelExampleAttacker's Effort to Evade
Hash valuesA hash matching a known malicious fileTrivial — change one byte, hash changes entirely
IP addresses192.168.1.1Easy — switch to a different IP
Domain nameswww.malicious-example.comSimple — register a new domain
Network artifactsA suspicious User-Agent stringModerate — requires tooling changes
Host artifactsA malware-created filenameAnnoying — requires rebuilding malware components
ToolsJohn the Ripper password crackerChallenging — requires finding/building new tools
TTPs (Tactics, Techniques, Procedures)An attacker's entire behavioral playbookHardest — requires fundamentally changing how they operate
📖DefinitionDEF-522

Tactics: High-level overview of attacker behavior. Techniques: Detailed descriptions of behavior relating to a tactic. Procedures: Highly detailed descriptions of a technique — together, TTPs are the hardest indicators to detect and defend against.

Triage & the Containment, Eradication, and Recovery Phase

Log Formats: Syslog, JSON, XML, CSV & CEF

📖DefinitionDEF-529

Syslog: Both a standard for message logging and a protocol enabling systems to send event messages across a network to a designated syslog server, widely used for consolidating logs from multiple sources.

Syslog FormatCopy
<34>1 2024-03-11T14:22:03Z host-01 sshd 1234 - Failed password for root from 203.0.113.9
Fields, in order: Priority value (facility + severity), version, timestamp, hostname, application name, process ID, message ID, and the structured message itself.
JSON Log FormatCopy
{
  "timestamp": "2024-03-11T14:22:03Z",
  "host": "host-01",
  "event": "auth_failure",
  "user": "root",
  "src_ip": "203.0.113.9"
}
What this is: JavaScript Object Notation — human-readable, uses key-value pairs, and is widely used by web applications and APIs due to its flexibility and easy parsing by nearly every programming language.
XML Log FormatCopy
<event>
  <timestamp>2024-03-11T14:22:03Z</timestamp>
  <host>host-01</host>
  <user>root</user>
</event>
What this is: Extensible Markup Language — uses nested tags to structure data, common in enterprise systems and older SIEM integrations, more verbose than JSON but strictly hierarchical.
CSV Log FormatCopy
timestamp,host,user,src_ip
2024-03-11T14:22:03Z,host-01,root,203.0.113.9
What this is: Comma-Separated Values — simple tabular format, easy to import into spreadsheets and SQL databases, but lacks native support for nested/complex data structures.
CEF Log FormatCopy
CEF:0|Acme|Firewall|1.0|100|Failed Login|5|src=203.0.113.9 duser=root
What this is: Common Event Format — a standardized format specifically designed for security devices, used widely across SIEM platforms. Fields: CEF version, vendor, product, version, signature ID, name, severity, then extension key-value pairs.
🎯Exam TipEXM-503

Recognize each format by its shape even without labels: Syslog starts with a bracketed priority value; JSON uses curly braces and key-value pairs; XML uses nested angle-bracket tags; CSV is flat and comma-delimited; CEF always begins with the literal string "CEF:" followed by a pipe-delimited header.

HIDS vs. NIDS

Comparison — Host-Based vs. Network-Based IDS
TypeMonitorsDeployment
HIDS (Host-based IDS)A single host/device — file integrity, system calls, local logsInstalled directly on the endpoint
NIDS (Network-based IDS)Network traffic across multiple hostsDeployed at strategic network points (e.g., just inside a firewall)
🛡️Blue Team TipBT-502

HIDS and NIDS are complementary, not competing — a NIDS placed at the network perimeter won't see malicious activity happening entirely within an already-compromised host's local file system, and a HIDS on one machine can't see network-wide traffic patterns. Mature security programs deploy both.

Signature-Based vs. Anomaly-Based Detection

Comparison — Signature-Based vs. Anomaly-Based Detection
MethodMechanismStrengthWeakness
Signature-basedMatches traffic/files against a database of known malicious patternsLow false positive rate for known threatsCannot detect novel/zero-day threats without an existing signature
Anomaly-basedCompares activity against an established baseline of normal behaviorCan detect novel, previously unseen threatsHigher false positive rate, since legitimate but unusual behavior can trigger alerts

Suricata Rules & Logs

📖DefinitionDEF-530

Suricata: A free and open-source, signature-based NIDS/NIPS engine that inspects network traffic using externally developed rulesets, and can also function as an IPS by actively blocking matched traffic.

Suricata Rule SyntaxCopy
alert tcp any any -> $HOME_NET 22 (msg:"Possible SSH brute force"; flow:to_server; threshold:type both, track by_src, count 5, seconds 60; sid:1000001; rev:1;)
Breaking this down, left to right: alert is the action; tcp is the protocol; any any is the source IP/port; -> indicates direction; $HOME_NET 22 is the destination network variable and port; everything in parentheses is the rule options block, containing the message, flow direction, a threshold to reduce noisy repeated alerts, a unique signature ID (sid), and a revision number.
Core Suricata Rule Components
ComponentFunction
ActionWhat happens on a match: alert, drop, pass, reject
HeaderProtocol, source/destination IP and port, direction operator
Rule optionsEverything in parentheses — msg, sid, rev, flow, threshold, and content matching
Suricata Alert Log (EVE JSON)Copy
{
  "timestamp": "2024-03-11T14:22:03.112",
  "src_ip": "203.0.113.9",
  "dest_ip": "10.0.4.12",
  "dest_port": 22,
  "alert": {
    "signature": "Possible SSH brute force",
    "severity": 2
  }
}
What this shows: Suricata's EVE JSON output format logs matched alerts with full context — source/destination, the matched signature's message, and severity — directly consumable by a downstream SIEM tool.

SIEM Log Ingestion

Recall the three-step SIEM process from Module 1: collect, normalize, analyze. Normalization is precisely what allows a SIEM tool to compare a Syslog entry from a firewall against a JSON entry from a cloud application — translating both into one consistent internal schema.

🛰️SOC InsightSOC-502

A SIEM ingesting Suricata's EVE JSON alerts alongside Syslog firewall logs and CEF-formatted endpoint logs is a completely typical real-world setup — the whole point of the SIEM is to make these differently-shaped data sources queryable together in one place.

Querying with Splunk (SPL) & Chronicle (YARA-L/UDM)

Splunk Query (SPL)Copy
index=security_logs sourcetype=suricata alert.severity<=2
| stats count by src_ip, alert.signature
| sort -count
What this does: Searches Suricata alerts with severity 2 or lower (higher priority), counts occurrences grouped by source IP and signature, sorted by frequency.
Real SOC usage: This kind of query quickly surfaces which sources are triggering the most high-priority alerts — a natural starting point for daily triage.
📖DefinitionDEF-531

Unified Data Model (UDM): Chronicle's standardized schema that all ingested logs are normalized into, regardless of original source format — enabling consistent searching across every log type.

YARA-L: Chronicle's rule-writing language for detection logic, inspired by the YARA pattern-matching tool used widely in malware research.

Chronicle Search (UDM)Copy
metadata.event_type = "NETWORK_CONNECTION"
AND target.port = 22
AND security_result.severity = "HIGH"
What this does: Searches Chronicle's normalized UDM fields for high-severity network connection events targeting port 22 (SSH) — functionally similar to the Splunk query above, but expressed through Chronicle's UDM field naming convention rather than raw source fields.
🎯Exam TipEXM-504

Both queries above answer a nearly identical question — "show me high-priority SSH-related alerts" — using two different platforms' native syntax. Interviewers sometimes ask you to translate a plain-English request into whichever platform's syntax they use; understanding the shared underlying logic (filter → aggregate → sort) matters more than memorizing exact syntax for a specific vendor.
