Chapter 03 · Remote Access Tools

Reaching a server from your own desktop

Servers usually sit somewhere else — another machine, another building, the cloud. This chapter walks through the everyday Windows tools engineers use to log into a remote Linux machine and run commands on it as if it were right in front of them: PuTTY, the built-in ssh command, Git Bash, and MobaXterm — plus the network setup that makes the connection actually work.

01

Why do we need remote access tools?

The server you want to use is usually not the machine in front of you

Where this comes from: The lecture opens by saying: in this section we'll look at some useful tools you should know about. The scenario it draws is simple — "imagine there's a server on the right-hand side; how can we access it?" That is exactly what we'll practise. from transcript
1 · The situation

You are sitting at your own Windows desktop or laptop. The machine you actually want to work on — a Linux server or virtual machine — is somewhere else. You need a way to open its command line from your machine, type commands, and see the results, without physically standing at that server.

2 · What we'll practise

In this chapter we set up and use several tools that all do this same job. We'll use two virtual machines as our "remote servers": a CentOS VM and an Ubuntu VM (which is Debian-based). Everything you learn here applies identically to a real remote server on the internet.

02

Installing PuTTY

A tiny, classic SSH client for Windows

1 · Download it

Go to the PuTTY website and click the Download button. The transcript points you to the official site — putty.org — to grab the installer.

Small correction: the audio says "google.org", but the tool is actually downloaded from putty.org (the official PuTTY site). Use putty.org, not google.org. clarified
2 · Install it

Download the MSI installer (the standard Windows installer package). The steps are simple: run it, click through, and Finish. Once done, PuTTY is installed.

3 · Open it

Go to the Windows Search, type PuTTY, and open it. The first time, you'll see the PuTTY configuration screen — a window asking for a host name and port.

What is an MSI? An .msi file is Microsoft's standard installer format on Windows. Double-clicking it launches a guided install (Next → Next → Finish) that copies the program onto your system and registers it. expanded
13 · Summary

PuTTY is installed from putty.org via a simple MSI installer, then opened from Windows Search. Before we can connect, though, we need one piece of information: the server's IP address.

03

Finding the server's IP address

You connect to a machine by its IP — so first you must read it

1 · Why we need the IP

To reach any machine over the network you need its IP address — its unique address on the network. So the first thing we do on the CentOS VM is open a terminal and read its IP.

2 · On CentOS — the ifconfig command

On the CentOS virtual machine, open the terminal and run:

ifconfig
Command on CentOS to display network interfaces
3 · Reading the output

In the output you'll see a network interface that shows the flags BROADCAST RUNNING MULTICAST. Next to that interface is an IP address. That is the IP we use to access this server.

What is a network interface? It's the machine's connection point to a network (wired or virtual). A single machine can have several. The one marked RUNNING with an IP is the active one you connect through. expanded
🏠
Real-life analogyAn IP address is the server's street address. Before you can post a letter (open a connection), you must first look up and copy down that address.
13 · Summary

On CentOS, ifconfig shows the network interface and its IP address (the one with BROADCAST RUNNING MULTICAST). Copy that IP — we'll type it into PuTTY next. But the very first attempt will fail, which teaches us something important.

04

The connection error — and the Bridged Adapter fix

Why the first attempt fails, and the VirtualBox setting that repairs it

1 · The first attempt

Open PuTTY. In Host Name (or IP address) type the IP you read from the VM. Leave the Port at its default, 22. Click Open to connect.

2 · It fails

The connection errors out. In the transcript this is expected — the default VirtualBox networking (NAT) doesn't let your host PC reach the VM directly by that IP. We need to change one VirtualBox setting.

3 · Change the network to Bridged Adapter

In the running VM's window go to Devices → Network → Network Settings. By default the adapter is Attached to: NAT. Change that to Bridged Adapter and click OK.

Devices
→
Network
→
Attached to: Bridged Adapter
Fig 4.1 — Switching the VM's network adapter from NAT to Bridged Adapter
4 · Reboot so the change takes effect

Check the IP again with ifconfig — it's still the same, because the change hasn't applied yet. So we restart the VM. In the VM terminal run:

reboot
Restarts the virtual machine so the new network setting applies
5 · A new IP appears

After the VM restarts, log back in, open the terminal, and run ifconfig once more. This time you get a new IP — in the lecture's case 192.168.1.35 (yours will differ). A new, bridged IP means the network is now set up correctly and PuTTY should work.

NAT vs Bridged, in one line: With NAT the VM hides behind the host and isn't directly reachable. With a Bridged Adapter the VM gets its own address on your real home network — so your PC (and PuTTY) can reach it directly, just like any other device. expanded
✗Changing the setting to Bridged Adapter and expecting it to work immediately.
✓Reboot the VM after changing the adapter — the new IP only appears after the restart.
13 · Summary

The first PuTTY attempt fails because the VM uses NAT. Switching the VirtualBox adapter to Bridged Adapter and rebooting gives the VM a directly reachable IP. Now we can actually log in.

05

Connecting with PuTTY

Host key, login name, password — and you're in

1 · Enter the new IP

Open PuTTY again. In Host Name (or IP address) type the new bridged IP (e.g. 192.168.1.35), keep Port 22, and click Open.

2 · Accept the security certificate (host key)

The first time you connect, PuTTY shows a security alert about the server's key. This is normal for a first connection — simply Accept it. PuTTY then remembers this server.

3 · Log in

You now see login as:. Enter your username — in the lecture the user is paul — then enter that user's password. On success you're connected: you now have the VM's terminal open inside PuTTY.

Why a "host key"? The first time you connect, the server presents a unique fingerprint (its host key). Accepting it lets PuTTY recognise the same server next time and warn you if it ever changes — a protection against connecting to an impostor. expanded
4 · Customise the look (optional)

Inside the PuTTY session you can go full screen and change settings. Right-click the title bar → Change Settings → Appearance, and change the font and size, then Apply. It's the same terminal as the VM — commands like pwd and hostname all work exactly as they do locally.

🪟
Real-life analogyPuTTY is a window cut into the remote machine. Everything you type goes through the window to the server; everything the server prints comes back through it. The glass (fonts, colours) is yours to decorate.
13 · Summary

Enter the bridged IP, accept the host key, log in as your user with a password, and you have a live remote terminal in PuTTY. Next we look at how the commands differ between CentOS and Ubuntu.

06

CentOS vs Ubuntu — small command differences

ifconfig vs ip addr, and the same Bridged + reboot dance

1 · Repeat on the Ubuntu VM

Now we do the same for the second (Ubuntu) VM. Open its terminal. Because Ubuntu is Debian-based, the command to read the IP is slightly different — instead of ifconfig, use:

ip addr
The IP-address command on Debian/Ubuntu systems
2 · Same pattern in the output

The output shows the same pattern — interfaces with flags like BROADCAST MULTICAST, and an IP address. But you already know this first IP won't work until we bridge the network.

3 · Same fix — Bridged Adapter + reboot

Go to Devices → Network Settings, switch from NAT to Bridged Adapter, click OK, then run reboot. After the restart, run ip addr again — in the lecture the Ubuntu VM now shows 192.168.1.16, confirming the network is set up.

The workflow is identical across distros; only the IP command changes (ifconfig vs ip addr). But when we try PuTTY against the Ubuntu VM, we hit a new error — because a service is missing.

07

Starting the SSH server so logins are accepted

A machine can only accept remote logins if its SSH service is running

1 · The new error

We enter the Ubuntu VM's IP into PuTTY and get an error again. The reason: to accept remote connections, a machine needs an SSH service enabled and running. The Ubuntu VM doesn't have it yet, so we install it.

2 · Install the OpenSSH server

In the Ubuntu VM's terminal, run:

sudo apt install openssh-server
Installs the package that lets this machine accept SSH logins
3 · Start the service

Installing the package isn't enough — we must start the service so it's actually listening. Basically, we're enabling a service so the server can be reached remotely from outside. Run:

sudo systemctl start ssh
Starts the SSH service (it may ask for the user's password, e.g. paul)
4 · Check it's running

Confirm the status with:

sudo systemctl status ssh
Should report the service as active (running)
What is a "service" here? A service (or daemon) is a background program that stays running to handle requests. The SSH service listens on port 22 for incoming logins. No running SSH service = no remote access, which is exactly the error we saw. expanded
🚪
Real-life analogyInstalling openssh-server builds the door. systemctl start ssh is what actually opens that door and posts someone to answer knocks. Without starting it, the door exists but stays locked.
5 · Now PuTTY works

With the SSH service active, open PuTTY again for the Ubuntu VM. This time it connects — accept the host key, enter the username and password, and you're in. The transcript even verifies it with cat /etc/os-release to prove it really is the Ubuntu VM.

13 · Summary

Remote logins require a running SSH server. On Ubuntu: sudo apt install openssh-server, then sudo systemctl start ssh (check with status). PuTTY is only the first client — next, three more ways to make the same SSH connection.

08

SSH straight from the Windows Command Prompt

No extra tool needed — ssh is built into modern Windows

1 · The idea

You don't always need PuTTY. Windows ships with an ssh command in its Command Prompt (CMD). First read the server's IP (e.g. ifconfig on CentOS), then open CMD.

2 · The command

SSH stands for Secure Shell. The pattern is ssh, then the username, then @, then the host name or IP. For user paul:

ssh paul@192.168.1.35
Syntax: ssh <user>@<ip-or-hostname>
3 · Same prompts as PuTTY

Press Enter. The first time it asks you to confirm the server's fingerprint — type yes. Then it asks for the password. Enter it, and you're logged into the VM straight from CMD. The same command works against a real remote server.

Memory trick
ssh = Secure SHell. Read it as "shell into that machine, securely." ssh user@address.
13 · Summary

Modern Windows has SSH built in: ssh paul@<ip> in CMD, confirm the fingerprint, enter the password. Same result as PuTTY, zero install. Next: Git Bash.

09

Git Bash

The terminal you install for Git can run ssh too

1 · What it is

Another option is Git Bash — the tool many people already have installed because they use Git. It gives you a Unix-style shell on Windows, and it can access a remote server the same way.

2 · Install it

Run the Git installer, double-click to launch, accept the licence, and click Next → Next → Next → Finish. Very simple steps.

3 · Use the same ssh command

Open Git Bash and use the identical SSH command you used in CMD:

ssh paul@192.168.1.35
Git Bash accepts the very same ssh syntax
Why does the same command work everywhere? PuTTY, CMD, and Git Bash are all just clients speaking the same SSH protocol to the server. The tool is a matter of taste — the underlying connection is identical. expanded
13 · Summary

Git Bash is a familiar Unix-style shell on Windows that runs ssh user@ip just like CMD. One last, more feature-rich tool remains: MobaXterm.

10

MobaXterm — a multi-tab terminal

A colourful client that keeps many server sessions side by side

1 · What makes it different

MobaXterm is a multi-tab terminal. Its big advantage: you can keep several remote sessions open at once, side by side, and switch between them easily. Its presentation is colourful and polished, which many people find nicer to work in — especially for reading files and logs.

2 · Open a session

Launch MobaXterm and go full screen. To connect to any terminal, click Session. There are several options; we use SSH. Actually the transcript calls it the Remote host field — put the server's IP there.

3 · Fill in the details

Tick Specify username and enter paul. Leave the Port at 22, click OK, accept the keys/permission prompt, then enter the password. If it asks to save the password, you can choose No. You're connected.

4 · Two servers, side by side

Now open a second session the same way for the other VM (e.g. copy its IP with ip addr, then Session → SSH → paste IP → username paul → password). Both terminals are now open side by side. That's the point: you can multitask, switching easily between one server's terminal and another's.

🖥️
Real-life analogyPuTTY gives you one window per server. MobaXterm is more like a control desk with several monitors in one frame — all your servers visible at once, a glance apart.
