Chapter 02 · Virtualization & Virtual Machines

One physical machine, many computers running inside it

This chapter builds, from first principles, one of the most commonly used ideas in the IT industry: how a single laptop or server can be split into several independent "computers" — each running its own operating system — using virtualization, a hypervisor, and virtual machines.

01

The Layers of a Computer

Prerequisite — you must see these layers before virtualization makes sense

Where this comes from: The lecture opens with a simple use case — "suppose you have a laptop or desktop" — and lists its components. We expand that opening into a clear layered model, because every later idea (virtualization, hypervisor, VM) is described as a layer added on top of these. from transcript, expanded
1 · The starting picture

Take any ordinary laptop or desktop. The lecture asks us to look at its components in order, from the bottom up. There are three layers stacked on each other.

2 · Layer one — Hardware

At the very bottom is the hardware: the physical parts you could touch. The transcript names RAM, CPU, and hard disk / storage. These are the real, physical resources — the machine's actual "muscle."

2b · What are RAM, CPU, and hard disk?

CPU (Central Processing Unit) is the "brain" that does the calculations. RAM (Random Access Memory) is fast, temporary working memory the CPU uses while programs run. The hard disk is permanent storage that keeps your files even after the machine is switched off.

3 · Layer two — Operating System

On top of the hardware sits an operating system (OS) — for example Windows. As covered in Chapter 1 of this course, the OS is the software that lets applications talk to the hardware.

4 · Layer three — Applications & the people who build them

On top of the OS run applications. The lecture gives a concrete scenario: suppose you are a software engineer who builds an application, and this software has target users who will run it.

Applications — the software an engineer builds & ships
Operating System — e.g. Windows
Hardware — RAM, CPU, Hard Disk / Storage

Real-life analogyThink of a building: the foundation is the hardware, the structure and utilities are the operating system, and the offices and people working inside are the applications and their users. Each layer rests on the one below it.

13 · Summary

A normal computer is three stacked layers: hardware, then one operating system, then applications. The catch is that this gives you exactly one operating system per machine. The next chapter shows why that single-OS limit becomes a real problem.

02

The Problem: needing more than one OS

Why "one machine, one operating system" isn't enough

1 · The problem the lecture raises

Our software engineer built an app, and it has target users. But those users don't all run the same operating system — some are on Windows, some may be on other operating systems. So the engineer needs to test the application on more than one OS to be sure it works for everyone.

2 · Why this is hard the naive way

Your machine already has Windows installed as its single OS. To test on a different OS the obvious options are painful: buy a second physical computer, or wipe and reinstall a different OS over your current one — losing your existing setup.

3 · What happens if we don't solve this

Without a better option, every extra operating system you want to try means extra hardware to buy, more desk space, more cost, and a real risk of breaking the primary OS you depend on every day.

🚪
Real-life analogyImagine you only have one room, but you need to host three separate private meetings at the same time. Buying three more houses is wasteful. What you really want is a way to partition the single room into three sealed, independent rooms — instantly, and for free. That is exactly what the next idea gives us.
13 · Summary

We need to run several different operating systems on the same physical machine, safely and cheaply, without buying new hardware or endangering the OS we already use. The solution is a new layer called virtualization.

03

What is Virtualization?

The extra layer that lets one machine run many operating systems

5 · Simple explanation (from the transcript)

The lecture says: on top of your existing operating system (e.g. Windows), a new layer of virtualization is added. On top of that layer, you can set up whatever operating system you require — and not just one, you can set up multiple different operating systems at the same time.

6 · Technical explanation

Virtualization is the technique of using software to create "virtual" versions of a computer on top of one real, physical computer — so a single machine can run several independent operating systems together, each believing it has a computer to itself.

2 · So what actually is that layer?

Naturally the next question is: how does this virtualization layer actually work — what is that layer, in reality? The transcript answers with a single word, and it is the heart of this whole topic: the hypervisor. That's the next chapter.

Memory trick

13 · Summary

Virtualization is a software layer placed above your host OS that lets one physical machine run many operating systems at once. To understand how that layer really works, we need to meet the software that is that layer — the hypervisor.

04

What is a Hypervisor?

The software that creates and runs virtual machines

5 · Simple explanation (from the transcript)

The virtualization layer, in reality, is a piece of software called the hypervisor. The transcript's exact definition: "A hypervisor is a software that creates and runs virtual machines."

6 · What it does for you

On top of your main machine, the hypervisor is what actually sets up virtualization, builds the virtual machines, and maintains them. Without a hypervisor, there is no virtualization layer at all.

7 · The simple use case

Practically: on your main machine you simply install the hypervisor (it's just a software), and with its help you create separate virtual machines — and inside each one you can set up different types of operating systems, one or many.

13 · Summary

A hypervisor is the software that creates and runs virtual machines — it is the virtualization layer. VirtualBox is a free, open-source example. Next we open the hood: how does the hypervisor actually give each VM its resources?

05

How a Hypervisor Works — Sharing Resources

Dividing the host's real RAM & disk into smaller "mini computers"

1 · The question

What does the hypervisor do that lets us use different operating systems? The transcript's answer in one line: it simply shares hardware resources from the host operating system.

5 · What "shares hardware resources" means

The transcript walks through a diagram, and we'll follow it exactly. Your main computer has real resources — the example given is 8 GB of RAM and 100 GB of hard disk. You install a hypervisor on it (e.g. VirtualBox).

7 · Internal working — the division

The hypervisor takes those available resources (8 GB & 100 GB) and divides them to create small "mini computers." In the transcript's example it carves out one mini computer with 2 GB RAM + 20 GB disk, and another with 2 GB RAM + 10 GB disk. Each mini computer is a virtual machine.

Important point from the transcript: the resources inside a VM are real, actual resources — they are genuinely shared out from the main computer's hardware, not imaginary. That's why the two VMs' RAM (2 GB + 2 GB) is taken from the host's real 8 GB. from transcript
🍕
Real-life analogyThink of the host's resources as one large pizza (8 GB). The hypervisor is the person slicing it: it hands a 2-slice portion to one guest and a smaller portion to another. Each guest gets real pizza — just a divided share of the same original pizza.
Memory trick
Hypervisor = a divider. Big real machine in, several small real-resourced machines out.
13 · Summary

A hypervisor works by sharing and dividing the host's real hardware resources into smaller mini computers — virtual machines — each with a genuine slice of RAM and disk. But a mini computer with only RAM and disk still can't do anything useful yet. It needs something on top, which is the next chapter.

06

What is a Virtual Machine?

A software-made "mini computer" that needs its own OS & runs isolated

5 · Why it's called "virtual"

The mini computer the hypervisor built is called a virtual machine (VM). The transcript explains the name plainly: there is no actual, separate physical machine there — it's made purely with the help of software. Because it isn't a real physical box, we call it a virtual machine.

6 · A VM needs its own operating system

A virtual machine, on its own, won't work without an operating system — just like any real computer. So on top of each VM we set up another operating system. This is why you can end up with, say, Windows as the host and a different OS running inside a VM.

10 · Real-world benefits (from the transcript)

To learn or use a new operating system, you don't need to buy a new machine. You can easily test and try your application on different operating systems, at no extra hardware cost.

11 · The key benefit — isolation

The transcript stresses this as an important point: every VM runs in its own isolated environment. So there is no risk of any issues to your primary operating system — whatever happens inside a VM (a crash, a bad install, even a virus) stays inside that VM and doesn't affect the host or the other VMs.

13 · Summary

A virtual machine is a software-created mini computer with a real slice of the host's resources; it needs its own operating system, and it runs isolated so it can't harm your primary OS. Next, an important distinction: where the hypervisor itself sits — on top of an OS, or directly on the hardware.

07

Type 2 — the Hosted Hypervisor

A hypervisor that runs on top of a normal host OS

5 · What we've seen so far is Type 2

The whole example up to now — installing VirtualBox on top of your existing Windows — is a hosted hypervisor, also called Type 2. The transcript notes the host OS in the example (it points to something like VMware Workstation running over the host OS) sits on top of the primary OS.

6 · Why it's called "hosted"

It's called hosted because the hypervisor is used on top of the host / primary operating system. There is a full normal OS underneath it (like Windows), and the hypervisor is just another program running on that OS.

13 · Summary

A Type 2 / hosted hypervisor (like VirtualBox) runs as software on top of a normal host OS. It's the easiest to start with. But there's another kind that removes the host OS entirely — Type 1 — covered next.

08

Type 1 — the Bare Metal Hypervisor

A hypervisor installed directly on the hardware, with no host OS

5 · Simple explanation (from the transcript)

Type 1, also called a bare metal hypervisor, is set up (installed) directly on the hardware. It does the same job — creating separate virtual machines by sharing resources — but the key difference is: there is no host operating system underneath it.

2 · The common confusion the transcript addresses

People ask: without any operating system, how can you even install a hypervisor onto hardware? The transcript's answer: a Type 1 / bare metal hypervisor works like a mini operating system itself. Whatever basic requirements are needed to set up on hardware are already built into the hypervisor, so it doesn't need a separate host OS.

10 · How clouds & big orgs use it

Cloud providers own very large infrastructure and servers with lots of resources. Using Type 1 hypervisors they cut this into many small VM segments and simply assign them to different users or organizations. Some organizations also do this in-house: they take one very large machine and divide it into small VMs for different roles, teams, or kinds of work.

A Type 1 / bare metal hypervisor installs straight onto hardware and behaves like its own mini OS, with no host OS below it. It powers data centers and clouds like AWS, which slice giant machines into many VMs for different customers. Finally: why do companies choose all this in the first place?

09

Why Companies & Clouds Use Virtualization

Less headache, cheaper, and scalable

10 · Reason one — less maintenance headache

The transcript's first point: companies get no headache of maintaining and managing the underlying machines. When you use a cloud/virtualization provider, they handle the heavy tasks for you — security, keeping operating systems updated, and all the many maintenance chores that come with running physical servers.

10b · Reason two — cheaper

Virtualization is cheap. Instead of buying a full physical machine for every need, you share one machine's resources across many VMs — far less hardware to purchase and power.

10c · Reason three — scalable

It is scalable. If you need to increase resources — more RAM, more storage — you can simply do that when the need grows, without buying and physically installing new hardware each time.

5 Interview Questions

What is virtualization?
A technique that uses software to run several independent operating systems on one physical machine at the same time, by adding a virtualization layer (a hypervisor) above the hardware/host OS.
What is a hypervisor?
Software that creates and runs virtual machines. It divides the host's real hardware resources into smaller mini computers, and manages them. Examples: VirtualBox, VMware vSphere.
What is a virtual machine, and why is it "virtual"?
A software-created mini computer that gets a real, divided share of the host's RAM and disk and needs its own guest OS. It's called "virtual" because there is no separate physical machine — it exists only through software.
What's the difference between a Type 1 and a Type 2 hypervisor?
Type 2 (hosted) runs on top of a normal host OS, like VirtualBox on Windows. Type 1 (bare metal) installs directly on the hardware with no host OS and acts as its own mini OS — used in data centers and clouds (VMware vSphere, Citrix XenServer, AWS).
Why do companies and cloud providers use virtualization?
It removes the maintenance headache (the provider handles security and OS updates), it's cheaper than buying many physical machines, and it's scalable — resources can be increased on demand. VMs also run isolated, protecting each other.
