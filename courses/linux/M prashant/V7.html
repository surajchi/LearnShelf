01

Files & Directory Management

Reading, editing, creating, moving and listing — the daily bread

Problem · why start here

Whatever your role — support, testing, or development — you spend most of your Linux day looking at files: reading a log, editing a config, creating a folder, or checking what's in a directory. The lecture starts with this group because every other group builds on it.

🗄️
Real-life analogyThe filesystem is a giant filing cabinet. cd is walking to a drawer, ls is opening it, less is reading a document without taking it out, cp/mv are photocopying or relocating it, and rm is the shredder. pwd asks "which drawer am I standing at?"
1 · Reading big files — less and more

Sometimes you have a command's output, sometimes a file — and you want to read it page by page instead of it flying past. less file and more file both do this. The lecturer's personal pick is less, used most of all for big files, because inside it you can search forward and reverse very easily.

Key inside lessAction
/wordForward search (top → bottom)
?wordReverse search (bottom → top)
n / NNext match / previous match
Shift+G / gJump to last line / first line
qQuit
less vs more: both page through content, but less gives you easy forward and reverse searching, which is why it wins for large log files. from transcript
2 · Editing files — vi, vim, nano

To create or edit a file you need a text editor. The classic is the vi editor; you'll also see vim (vi improved) and nano. vi file or nano file opens it. In vi: press i to start typing, Esc to stop, then :wq to save and quit. In nano: just type, then Ctrl+X, Y, Enter.

3 · Quick read — cat

cat file is the simplest of all: it dumps the whole file to the terminal. Perfect for a small file. For anything long, go back to less.

4 · Create & delete — mkdir, rm

mkdir foldername = make directory, when you need a new folder. rm file deletes a file; rm -r folder deletes a directory and its contents.

Care with rm -rf: there is no recycle bin in Linux — deletion is permanent. rm -rf force-deletes recursively with no confirmation, so always re-read the path before pressing Enter. best practice
5 · Copy, move & rename — cp, mv

cp source dest copies a file (add -r for a folder). To move a file or directory somewhere else, use the move command: mv file folder/. And there is no separate rename command — mv oldname newname is also how you rename.

6 · Where am I — pwd

pwd = present working directory. Whatever location you're working in, it prints the full absolute path. Use it whenever you need to check where you currently are.

7 · What's here — ls

ls is the listing command for any directory. With options it tells you far more: permissions, the owner, when the file was last modified, and its size. That's why ls -l (long listing) is what you'll actually type most days.

bash — daily file work
$ pwd
/home/paul
$ ls -lh
-rw-r--r-- 1 paul paul  220 Jul 16 10:20 notes.txt
drwxr-xr-x 2 paul paul 4.0K Jul 16 10:22 projects
     ^perms  ^owner ^size ^last modified  ^name
$ mkdir reports
$ cp notes.txt reports/
$ mv notes.txt notes-old.txt   # rename with mv
$ less /var/log/syslog       # /word to search, q to quit

Summary

02

Searching & Compression

find, grep, and the archive family — tar, gzip, gunzip, zip, unzip

Problem · needles and haystacks

A real server holds thousands or even lakhs (hundreds of thousands) of files. Two questions come up constantly: "where is that file?" and "which line inside this huge file mentions the error?" find answers the first, grep the second. Then, once you've found your data, you often need to ship it or back it up — that's compression.

1 · Find files — find

find searches a big server for a file. Its real power is conditional searching: not just by name, but by time (e.g. files older than two months) and by size (e.g. files larger than 10 GB). That's how you clean up a full disk.

bash — conditional find
$ find /var -name "*.log"          # by name
$ find /data -mtime +60             # older than 2 months
$ find /data -size +10G              # bigger than 10 GB
$ find /data -size +10G -mtime +60   # combine conditions
2 · Search inside files — grep

grep finds a pattern — a word or string — inside files. It is heavily used in log analysis and data analysis: you have one enormous file and need every line mentioning a particular word. grep "error" app.log. Add -i to ignore case, -r to search a whole folder, -n for line numbers.

huge log file
500,000 lines
→

→
just the 12
matching lines
Fig 2.1 — grep is the filter that turns an unreadable file into the handful of lines you care about.
3 · Why compress?

Two everyday reasons from the lecture. First, transfer: you have a very big file — 2 GB, 5 GB, 10 GB — and moving it as-is is slow. Second, backup: you need to preserve a lot of data on your server. In both cases you simply archive it (bundle many files into one) or compress it (make it smaller).

4 · The archive family — tar, gzip, gunzip, zip, unzip

tar ("tape archive") bundles a whole folder into one file, and with -z compresses it too: tar -czf backup.tar.gz folder/ (create, gzip, file). Reverse it with tar -xzf backup.tar.gz (extract). gzip file compresses a single file, gunzip file.gz decompresses it. zip/unzip handle the cross-platform .zip format.

bash — archive & compress
$ tar -czf backup.tar.gz /data/reports/  # folder -> one small file
$ tar -xzf backup.tar.gz                # extract it back
$ gzip bigfile.csv                       # -> bigfile.csv.gz
$ gunzip bigfile.csv.gz                  # -> bigfile.csv
$ zip -r out.zip folder/  &&  unzip out.zip
Common mistake: gzip compresses a single file only — it cannot bundle a folder. For a whole directory use tar (which is why tar -czf is so common). common mistake

Summary

find locates files (by name, time, or size), grep locates lines inside them, and tar/gzip/zip pack data up for transfer or backup. That closes the basic file operations. Next: the network.

03

Basic Networking Commands

Troubleshooting connectivity and pulling files across the wire

Problem · why this group matters

In a job the machine you work on is a server in a data centre or the cloud — not the laptop in front of you. So when an application misbehaves, half the questions are network questions: what is my IP, can I reach that server, is that port open, why won't this download? These commands are your basic network troubleshooting kit.

📮
Real-life analogyThe network is the postal system. Your IP address is your street address, ping is knocking to see if anyone's home, netstat is checking which of your own mailboxes are open, and telnet is checking whether one specific letterbox at that address accepts post.
1 · Download files — curl and wget

Both let you download a file from the internet: you have a download link, you pass it to the command. wget https://site.com/app.tar.gz is the simple downloader. curl is a bit more powerful — it supports more protocols and HTTP methods like GET and POST, so you can send data to and fro from a server, not just pull files. That's why curl is the tool of choice for talking to APIs.

2 · My address — ip and ifconfig

When you need the IP-related information of the server you're working on, or details about its network interfaces, use ip addr (modern) or ifconfig (older). Look for the inet line.

3 · Ports & connections — netstat and ss

Say you need details about ports or already-established connections — netstat is really helpful here. To check whether a port is open, ss does the same job with modern, faster tooling. The usual invocation is netstat -tulpn or ss -tulpn: tcp, udp, listening, process, numeric.

4 · Log into a server — ssh (and ftp)

ssh user@server = Secure Shell, used to access a remote server. The key word is secure: the connection is encrypted. FTP (File Transfer Protocol) is also useful for moving files, though for anything sensitive you'd prefer its secure cousins (sftp/scp).

5 · Can I reach it — ping and telnet

Both check connectivity from your server to a particular server. ping is the famous one: give it an IP and it tells you whether packets are getting through or being lost. telnet gives you a bit more detail: a server might be reachable overall, but is one particular port open and accessible on it? For that case telnet host port is more helpful.

📦
your serverremote server
Fig 3.1 — ping asks "is the host alive?" · telnet host port asks "is that one port open?"
bash — network troubleshooting
$ ip addr                       # my IP & interfaces
inet 192.168.1.20/24 ...
$ ping -c 3 10.0.0.5            # is host reachable?
3 packets transmitted, 3 received, 0% packet loss
$ telnet 10.0.0.5 3306          # is THAT port open?
$ netstat -tulpn              # my open ports + owners
$ ssh paul@10.0.0.5             # encrypted remote login
$ wget https://site.com/app.tar.gz
$ curl -X POST -d '{"id":1}' https://api.site.com

Summary

curl/wget pull data, ip/ifconfig show your address, netstat/ss show ports, ssh/ftp reach other machines, and ping/telnet answer "is it reachable, and is that port open?" Next: the programs running on the box.

04

Process Management Commands

Everything running is a process — see it, control it, stop it

Prerequisite · what is a process?

Whenever an application is running, a script is running, a command is running — everything is all about processes. Whatever work is happening on a Linux machine is happening in the form of a process. Linux gives each one a number called a PID (Process ID) so you can point at it precisely.

🏭
Real-life analogyThe CPU is a factory floor and each process is a worker with a badge number (the PID). ps is the supervisor's clipboard listing every worker on shift; kill is sending one of them home.
1 · Is it running? — ps

Suppose you started an application. How do you check whether it is running or not? You check its process — and that's the ps command. Each process it lists has a PID. In practice you pipe it into grep to find the one you care about.

2 · Control it — kill

Once you have the PID, you may need to control that process — terminate it or stop it. That's kill PID, which asks it to shut down gracefully. If it's frozen and ignores you, kill -9 PID forces it to stop.

bash — find it, then stop it
$ ps -ef | grep python
paul  4821  1  6.0  ...  python app.py
      ^PID
$ kill 4821        # graceful stop
$ kill -9 4821     # force, only if stuck
ps -ef | grep app
→

→
kill PID
→
still stuck?
kill -9 PID
Fig 4.1 — The universal process workflow: find the PID, ask it to stop, force only as a last resort.
CommandWhat it does
ps -efList all running processes with their PIDs
ps -ef | grep nameCheck whether your app is running; get its PID
kill PIDTerminate/stop a process gracefully

Memory trick
-9 is the "no more Mr. Nice guy" signal. Plain kill lets the program clean up; -9 gives it no chance. Try plain first.
Related (covered in chapter 9): top for a live process/CPU monitor, nohup to run a process in the background, and nice for priority. added for clarity
Summary

Everything running is a process with a PID. ps shows them, kill stops them. Next: the users who run them.

05

User Management Commands

Who exists, who you are, and how to borrow root power

Prerequisite · Linux is multi-user

Linux was built for many people to share one machine safely. Each person has a user account with a numeric UID, belongs to one or more groups (each with a GID), and one special account — root — can do absolutely anything.

🏢
Real-life analogyAn office building. Every employee has a keycard (user account), keycards belong to departments (groups), and root holds the master key. sudo is borrowing the master key for one door, one time, with your name in the log.
1 · Does this user exist? — id

Say you need to check whether a user has been created on the server. Run id username. It tells you which groups that user is part of, plus their UID (user ID) and GID (group ID). All that information in one line.

2 · Who am I right now? — whoami

whoami prints the presently logged-in user. Simple, and constantly useful after you've been switching accounts.

3 · Change your password — passwd

If you want to change your own password, run passwd. It asks for the current one, then the new one twice.

4 · Become another user — su

su - username = switch user. Use it when you need to perform some action as a different user. The - loads that user's full environment. exit brings you back.

5 · Borrow root power — sudo

Separately, there's sudo: use it when you need to perform any action requiring root privileges. sudo command runs just that one command as the administrator, asking for your password and logging it — so you never have to stay logged in as root.

bash — identity & privilege
$ whoami
paul
$ id riya
uid=1001(riya) gid=1001(riya) groups=1001(riya),27(sudo)
 ^UID          ^GID           ^groups she belongs to
$ passwd                    # change my own password
$ su - riya                 # switch user (exit to return)
$ sudo systemctl restart nginx  # one command as root
CommandWhat it does
id usernameDoes the user exist? Their UID, GID & groups
whoamiThe presently logged-in user
passwdChange your own password
su - usernameSwitch to another user (exit to return)
sudo commandRun one command with root privileges
Memory trick
su = switch user (become someone, whole session). sudo = switch user and do one thing (borrow power for a single command).
Summary

id checks a user exists and shows their groups/UID/GID, whoami tells you who you are, passwd changes your password, and su/sudo switch user or borrow root. Next: installing the software they need.

06

Application Management Commands

Installing packages and managing services

Problem · a concrete scenario

The lecture's example: your application uses a database, so you need to install SQL on the server. What do you do? On Linux you don't download an installer — you use a package manager, which fetches the software from a trusted repository along with everything it depends on.

📦
Real-life analogyA package manager is a warehouse with a great catalogue. Ask for "mysql-server" and it delivers it plus every part it needs, remembers your order, and can take it all back cleanly (uninstall) without leaving a mess.
1 · Install packages — yum, dnf, apt

Which one you type depends on your Linux distribution. yum and its successor dnf are for Red Hat based systems (RHEL, CentOS, Fedora). apt is for Ubuntu/Debian. Using these you install any software or package. All of them need sudo — installing software is an admin action.

bash — install SQL, both families
# Ubuntu / Debian
$ sudo apt update
$ sudo apt install mysql-server

# Red Hat / CentOS / Fedora
$ sudo yum install mysql-server
$ sudo dnf install mysql-server    # dnf = modern yum
2 · Which binary am I actually running? — which

Say you're using some command and you want to know, behind the scenes, which binary executable it's actually running. which command tells you the exact path. Useful when two versions of a tool are installed and you're not sure which one you get.

3 · Prerequisite · what is a service?

As you've seen on Windows too, many applications run as a service — quietly in the background, started automatically. Good examples: SQL, a web server, or the firewall. They're not something you open a window for; they just run.

4 · Manage services — systemctl

To manage those services, use systemctl: start, stop, restart, check status, view logs, and enable them (so they start automatically on boot).

bash — services & binaries
$ which python3
/usr/bin/python3
$ sudo systemctl status  mysql   # is it running?
$ sudo systemctl start   mysql
$ sudo systemctl stop    mysql
$ sudo systemctl restart mysql
$ sudo systemctl enable  mysql   # auto-start on boot
CommandWhat it does
sudo yum install XInstall a package (Red Hat / CentOS)
sudo dnf install XInstall a package (modern Red Hat / Fedora)
sudo apt install XInstall a package (Ubuntu / Debian)
which commandWhich binary executable is behind this command
systemctl status XIs the service running? recent logs
systemctl start/stop XStart / stop a service
systemctl restart XRestart a service
systemctl enable XStart it automatically on boot
Memory trick
Red hat wears a YUM/DNF hat · Ubuntu is APT. And every service verb is the obvious English word: start, stop, restart, status, enable.
Summary

yum/dnf/apt install software depending on your distro, which reveals the real binary, and systemctl manages background services. Next: the variables those applications read.

07

Environment Variable Commands

Basic, but many applications refuse to run without them

Problem · why applications need them

These are fairly basic things, but plenty of applications require environment variables to work. The typical example from the lecture: some Java-based applications need JAVA_HOME to be set, or they simply won't start. An environment variable is just a named value that the shell — and every program you launch from it — can read.

🗒️
Real-life analogyEnvironment variables are settings sticky-notes on a shared office fridge. Any program that walks up can read "the Java kit is in cupboard 3" (JAVA_HOME) without being told each time. Change the note and everyone who reads it afterwards gets the new value.
1 · See what's already set — env and printenv

The two starting commands. With env or printenv you can see whichever environment variables are currently set and already existing on the system. To read just one: printenv JAVA_HOME or echo $JAVA_HOME — the $ means "give me the value of".

2 · Set your own — export and source

If you want to set your own, use export NAME=value. That makes it available to programs you start afterwards. But it disappears when you close the terminal — so to make it permanent, put the export line into your startup file (~/.bashrc) and run source ~/.bashrc to reload it into the current session without logging out.

bash — the JAVA_HOME example
$ printenv | head              # what's already set?
HOME=/home/paul
USER=paul
PATH=/usr/local/bin:/usr/bin:/bin
$ export JAVA_HOME=/usr/lib/jvm/java-17   # this session only
$ echo $JAVA_HOME
/usr/lib/jvm/java-17
$ echo 'export JAVA_HOME=/usr/lib/jvm/java-17' >> ~/.bashrc
$ source ~/.bashrc             # reload — now it's permanent
export VAR=val
→

→

Fig 7.1 — export is temporary. ~/.bashrc + source is how it survives a logout.
Watch the spaces: it must be NAME=value with no spaces around the =. Writing NAME = value makes the shell think NAME is a command — a classic beginner error. common mistake

Summary

08

Console, Output & System Control

cat, clear, echo, redirection — plus poweroff & reboot

1 · Taking input — cat

cat appears again here, in its "console" role: it's like taking the inputs — reading content in and writing it out to the terminal.

2 · Tidy the screen — clear

clear wipes your terminal clean. Shortcut: Ctrl + L, no command needed.

3 · Print anything — echo

echo is simple: it prints whatever you want to the terminal. You'll use it constantly inside scripts too, to report what's happening. echo "Backup finished".

4 · Redirect into a file — > and >>

Beyond printing, if you have a small word or string you want to redirect into a file, you can do that with echo too. echo "hello" > file.txt writes it (overwriting the file), while echo "hello" >> file.txt appends to the end.

bash — echo & redirection
$ echo "Backup finished"
Backup finished
$ echo "hello" > notes.txt    # write (OVERWRITES the file)
$ echo "world" >> notes.txt   # append (adds to the end)
$ cat notes.txt
hello
world
$ clear                        # or press Ctrl+L
One > versus two >>: a single > wipes the file and writes fresh. Two >> append. Using > when you meant >> silently destroys the old contents. common mistake
5 · System control — poweroff, reboot

The lecturer is honest here: these are not frequently used on the job — you rarely restart a production server — but they're mentioned for completeness. sudo reboot restarts the machine, sudo poweroff shuts it down.

CommandWhat it does
cat fileRead content in / print it out
clear (Ctrl+L)Clear the terminal screen
echo "text"Print to terminal; heavily used in scripts
echo "x" > fileRedirect into a file (overwrite)
echo "x" >> fileRedirect into a file (append)
sudo rebootRestart the system (rarely used in prod)
sudo poweroffShut the system down
Summary

cat/echo move text in and out, >/>> redirect it into files, clear tidies up, and reboot/poweroff exist but are rare in a job. Next: the leftovers that are still genuinely useful.

09

Other Useful Commands

Not top-frequency, but still genuinely useful on the job

Framing · what this group is

Everything so far was chosen for frequency. That doesn't mean you never use anything else — this final group collects the commands that are still very useful, just less constant.

1 · Memory & disk — free, du, df

The lecture's scenario: you have an application on a server and you're testing it. One part of testing is asking how much memory is that application consuming? For RAM, use free -h. du = disk utilisation, the space used by one particular directory or file (du -sh /var/log). df -h shows how full each filesystem/partition is overall.

bash — resources
$ free -h                # RAM: total / used / free
              total   used   free
Mem:           7.7Gi  3.1Gi  2.2Gi
$ du -sh /var/log       # size of ONE folder
1.8G  /var/log
$ df -h                  # whole filesystem, use%
/dev/sda1   40G  26G  14G  66%  /
2 · Permissions — chmod

Why it matters: when you work on a server you're mostly working in a team, with many users created on it. To work on a file, you must have the proper permission. chmod is what you use to change or maintain those permissions — for example chmod +x script.sh to make a script executable.

3 · Text processing — awk, cut, sed

Sometimes you're doing data analysis or log analysis and need one particular piece of the data. These three carve text up: awk prints chosen columns, cut slices by character or field, sed finds and replaces. You also use them heavily in scripting.

bash — carve up text
$ awk -F, '{print $2}' data.csv   # 2nd column of a CSV
$ cut -d: -f1 /etc/passwd        # 1st field, ':' separated
$ sed 's/error/ERROR/g' app.log    # find & replace all
4 · Links — ln

ln creates links: hard links and soft links (symbolic links). A soft link is like a shortcut pointing at the real file: ln -s /path/to/real /path/to/shortcut.

5 · Background jobs — nohup

nohup command & runs a process in the background — and keeps it running even after you log out. Essential for long tasks on a remote server.

6 · Automation — crontab

Whatever task you have — a script, say — that you want to automate, you use crontab to create a job that runs it on a schedule. crontab -e edits your schedule, crontab -l lists it.

bash — links, background, schedule
$ ln -s /opt/app/v2 /opt/app/current   # soft link (shortcut)
$ nohup ./long-backup.sh &              # survives logout
$ crontab -e                           # edit scheduled jobs
0 2 * * *  /home/paul/backup.sh          # run daily at 02:00
$ crontab -l                           # list my jobs
7 · Create a file — touch

The lecturer flags this one as accidentally left out of the earlier group: touch filename creates a new (empty) file. It belongs with chapter 1, but it's here so you don't miss it.

8 · What did I type? — history

A great many commands have already been executed in your session. When you want to see an old command — which ones did I run? — use history. Combine it with grep: history | grep ssh.

  • 5gzip compresses a single file; use tar -czf for a whole folder.
  • 6ping asks "is the host up?"; telnet host port asks "is that port open?"
  • 7Everything running is a process with a PID: ps to find it, kill to stop it.
  • 8su switches user for a session; sudo borrows root for one command.
  • 9yum/dnf = Red Hat, apt = Ubuntu. systemctl manages background services.
  • 10export is temporary; ~/.bashrc + source makes a variable (like JAVA_HOME) permanent.

5 Interview Questions

What's the difference between ping and telnet when troubleshooting?
ping tests whether the host is reachable at all and whether packets are being lost. telnet host port goes one level deeper: the server may be perfectly reachable, but you need to know whether one particular port (say 3306 for MySQL) is open and accessible. So ping for host connectivity, telnet for port connectivity.
Difference between curl and wget?
Both download files from the internet given a link. curl is more powerful: it supports more protocols and HTTP methods like GET and POST, so you can send data to and fro from a server. That makes curl the tool for talking to APIs, while wget is the straightforward file downloader.
How do you check if your application is running, and stop it?
Everything running is a process. Use ps -ef | grep appname to check whether it's running and grab its PID. Then kill PID to terminate it gracefully; if it's frozen, kill -9 PID forces it. Force-killing skips cleanup, so it's a last resort.
An application needs JAVA_HOME. How do you set it permanently?
Check what exists with env/printenv. Set it with export JAVA_HOME=/path/to/jdk — but that dies when the terminal closes. To make it permanent, add that same export line to ~/.bashrc, then run source ~/.bashrc to load it into the current session without logging out.
How do you find files older than two months or larger than 10 GB?
That's find's conditional searching. find /data -mtime +60 finds files modified more than 60 days ago; find /data -size +10G finds files bigger than 10 GB. You can combine them: find /data -size +10G -mtime +60. This is the standard way to hunt down what's filling a disk.

Common Mistakes

✗"gzip can compress a folder."
✓gzip is single-file only; use tar -czf for a directory.
✗"ping tells me the port is open."
✓ping only proves the host is reachable; use telnet host port for the port.
✗"NAME = value sets a variable."
✓No spaces around = — it must be NAME=value.
✗"echo x > file adds a line to the file."
✓Single > overwrites the whole file; >> appends.
✗"rm moves files to a trash bin."
✓There is no trash — deletion is permanent. Re-read the path.
