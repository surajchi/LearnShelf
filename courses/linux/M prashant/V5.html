Chapter 05 · SCP

Copying files between servers, securely, with one command

WinSCP moves files with a mouse. SCP — Secure Copy Protocol — does the same job from the command line: one short command copies a file from your machine to a remote server, or between two servers, all encrypted over SSH. If you know cp, you already almost know scp.

01

What is SCP?

Secure Copy Protocol — file transfer that is encrypted and authenticated

Where this comes from: The lecture introduces SCP for the case where "you have two servers and need to move a file between them." SCP = Secure Copy Protocol. from transcript
1 · The name tells the story

Notice the word Secure in the name — just like SSH is Secure Shell, SCP is Secure Copy Protocol. "Secure" means the transfer is encrypted and uses authentication.

2 · What "encrypted" buys you

Because the transfer is encrypted, the file / data you send travels safely — nobody in between can read it. That's the whole point of SCP over plain, unencrypted copying.

3 · What it's used for

SCP is used for securely transferring files between a local and a remote host — or between two remote hosts. Different combinations are possible; with one command you copy a file from one server to another.

Server A
→
scp (encrypted, over SSH)
→
Server B
Fig 1.1 — SCP copies a file from one machine to another through an encrypted SSH channel
🔒
Real-life analogyPlain copy is a postcard — anyone handling it can read it. SCP is a sealed, tamper-proof courier envelope: only the sender and the intended server can open it.
Memory trick
SSH = Secure SHell (run commands). SCP = Secure CoPy (move files). Same "secure over SSH" idea.
13 · Summary

SCP is the Secure Copy Protocol: an encrypted, authenticated way to copy files between local and remote machines. Before using it, you need a few things ready.

02

What you need before using SCP

The details and connectivity that make a transfer possible

1 · You must know the other server's details

To copy to another server you need to know that server's connection details: its hostname or IP address, plus a valid username and password (your credentials) on that machine.

2 · There must be connectivity between them

The two servers must be able to reach each other over the network — for example on the same network, or otherwise allowed to connect. No connectivity, no transfer.

"Where do I get the username?" The lecture answers a common student question: the credentials must already exist on the target server. In a company, you ask the system administrator — state your requirement, get manager approval, and they create your credentials. from transcript
3 · The three things to gather

In the demo, the presenter reads the target's IP with ifconfig (e.g. 10.21.55.1), and already knows the username paul and its password. So: IP, username, password.

🌐 Host / IP

The target server's address, e.g. from ifconfig or ip addr.

👤 Username

A valid account on the target server (e.g. paul).

🔑 Password

That user's password — asked at transfer time (unless keys are set up).

13 · Summary

Have the target's IP/hostname, a real username + password, and network connectivity between the machines. With those, the command itself is tiny.

03

The SCP syntax

Just like cp, but with a remote destination

1 · The shape of the command

SCP mirrors the ordinary copy command cp. You write scp, then the source (the local path where your file is), then the destination (where it should go).

2 · The remote destination format

The destination is a remote location, written as username@host:path — the username, then @, then the hostname or IP, then a colon :, then the path to place the file.

scp <source-path> user@host:/destination/path
Basic SCP syntax — local source, remote destination
Read the colon carefully: everything after the : is the path on the remote machine. paul@10.21.55.1:/tmp means "user paul, host 10.21.55.1, put it in /tmp." expanded
Memory trick
cp source dest on one machine → scp source user@host:dest across machines. Add s, add user@host:.
13 · Summary

Syntax: scp <source> user@host:/path. Source and destination can each be local or remote. Let's push a file to another server.

04

Local → Remote: pushing a file to another server

Copy from centos-01 to centos-02

1 · The setup

The demo uses two CentOS servers: centos-01 (where we are) and centos-02 (the remote target). On centos-01 we make a working folder and some files:

mkdir scp-demo
→
cd scp-demo
→
touch file1 file2
Create a folder and two files to transfer
2 · Put some content in a file

Optionally add text, e.g. write hello into file1. Check where you are with pwd (inside scp-demo) — two files, file1 and file2.

3 · The transfer command

Task: send file1 to centos-02. We know the IP (10.21.55.1), the user (paul) and its password. Run:

scp file1 paul@10.21.55.1:/tmp
Copies file1 into /tmp on the remote server (the colon separates host from path)
4 · What you see

SCP prints a progress line: file1  100% with the size and speed (e.g. 8kB/s). 100% means the transfer completed.

5 · Verify on the other server

Move to centos-02, go into /tmp, run ls — file1 is there. Read it with cat file1 and you'll see hello printed, proving the content came across intact.

Will it ask for a password? Yes — normally SCP asks for the remote user's password before transferring. In the demo it didn't, only because the presenter had already exchanged SSH keys (public/private key) with that server, allowing password-less login. In your case, expect a password prompt. from transcript
📤
Real-life analogyThis is "posting" a file: you hand file1 to the courier addressed to paul@10.21.55.1, drop-box /tmp. The 100% line is the delivery confirmation.
13 · Summary

scp file1 paul@10.21.55.1:/tmp pushes a local file to a remote server's folder; a 100% line confirms success, and ls/cat on the target verify it. Now the reverse direction.

05

Remote → Local: pulling a file from another server

Copy a file from centos-02 back to your machine

1 · The scenario

Now the opposite: a file lives on the remote server (centos-02) and you want it on your local system. Yes — SCP does this too; you just swap which side is remote.

2 · Make a file on the remote server

On centos-02, in /tmp, create remotefile and add text:

touch remotefile
→
cat remotefile → "This is file from centos-02"
A file that currently exists only on the remote server
3 · Pull it — source is now the remote side

Back on centos-01, the source becomes the remote path and the destination is local. The lecture also adds -v (verbose) for extra debug output — optional. Command:

scp -v paul@10.21.55.1:/tmp/remotefile .
Pulls remotefile from the server into the current directory ( . )
4 · About the . and -v

The final . (dot) means "put it in the current directory" — you could give any other path instead. The -v flag floods the screen with detail; it's only useful for debugging what's happening. Without it, the output is just the clean 100% line.

scp paul@10.21.55.1:/tmp/remotefile .
Same pull without -v — shows 100%, 27 bytes, and the rate
5 · Verify locally

Run ls — remotefile is now here. cat remotefile prints "This is the file from centos-02", confirming the pull worked.

Memory trick
Push vs pull is just which side wears the user@host:. Remote on the right = push; remote on the left = pull.
13 · Summary

To pull, put the remote path first and a local path (often .) last: scp paul@10.21.55.1:/tmp/remotefile .. -v is optional debug output. So files can flow both ways. Next: copying more than one file at a time.

06

Copying many files & using wildcards

List several files, or use * to grab them all

1 · More than one file

The earlier example sent a single file (file1). You can list multiple files before the destination and all of them transfer:

scp file1 file2 paul@10.21.55.1:/tmp
Both file1 and file2 are copied to the remote /tmp
2 · Wildcards

You can also use a wildcard. Instead of naming file1 file2, use file* — it matches every file starting with "file":

scp file* paul@10.21.55.1:/tmp
The * matches file1, file2, and any other file* — all transferred
3 · Copy everything in a folder

If you need to transfer all the files in a location, just use * — SCP will try to copy everything (remotefile, and all the rest).

What is a wildcard? * is a shell pattern meaning "any characters." file* expands to every matching filename before SCP runs, so one command handles many files. expanded
13 · Summary

Copy several files by listing them (file1 file2) or by pattern (file*, or * for all). One command, many files. Finally, where this really shines in practice.

07

Real-world use cases

Jump servers, pulling remote files, and a nod to rsync

1 · The jump-server case

A very handy real case from the lecture: sometimes you don't have direct access to a server. You reach it through a jump server (also called a bastion server) — you log into one server, and from there log into the next. SCP is very helpful in that kind of hop-based setup.

2 · Fetching a needed file

It's equally useful the other way: if you just need to bring a file from some server to yours, SCP pulls it in one line.

3 · Related tool — rsync

The lecture mentions a related, more powerful tool: rsync, also used for transferring files (and efficient at syncing only what changed). There's a separate video for it on the channel.

🪜
Real-life analogyA jump server is the security desk in a locked building: you can't walk straight to the vault, but you're allowed through the front desk, and from there to the room you need.
13 · Summary

SCP shines for hop-based access via jump/bastion servers and for grabbing files off remote machines. For heavier, incremental syncing, look at rsync. That completes the SCP tour.
