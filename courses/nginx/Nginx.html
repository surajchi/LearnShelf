Preface

Nginx is one of those pieces of infrastructure that most engineers rely on daily and understand shallowly. You know it "serves stuff fast" and "sits in front of your app," but the moment an interviewer or an incident asks why — why does a worker process handle ten thousand connections without ten thousand threads, why does location matching pick the block it does, why did a reload just drop connections — the shallow model runs out.

This book was built one chapter at a time, on purpose. Each chapter stands as a complete unit: objectives, definitions, history, internals, diagrams, worked examples, common mistakes, and a spaced-repetition style revision pass. By the end, you should be able to reason about Nginx from the socket up, not just copy configuration blocks from memory.

Read it in order the first time. After that, use the sidebar to jump straight to whatever you're debugging.

How the Web Actually Works

Fig 1.1 — The trip a single request makes before Nginx ever sees it

Learning objectives

  • Trace a URL from address bar to rendered page, naming every hop
  • Explain DNS resolution, TCP's three-way handshake, and the TLS handshake at a level you could whiteboard
  • Distinguish HTTP/1.1, HTTP/2, and HTTP/3 at the transport level
  • Understand why "the web is stateless" matters for how servers like Nginx are built

Definition

The web is a client–server system built on a stack of agreed-upon protocols: DNS to turn names into addresses, TCP (or QUIC) to move bytes reliably, TLS to encrypt them, and HTTP to give those bytes meaning as requests and responses. Nginx lives at the HTTP layer, but everything below it shapes how it behaves.

Why it exists

None of this was inevitable. Early networked systems assumed trusted, stable links between known machines. The web needed something else: a way for any client to reach any server, by name, over an unreliable public network, with content that could be cached, redirected, and load-balanced along the way. Each protocol layer solves one piece of that problem so the layers above don't have to.

A brief history

Tim Berners-Lee's original 1989 proposal paired three ideas that still hold: a naming scheme (URLs), a markup format (HTML), and a transfer protocol (HTTP). HTTP/1.0 opened and closed a new TCP connection for every request — brutally wasteful. HTTP/1.1 (1997) added persistent connections and pipelining. HTTP/2 (2015) multiplexed many requests over one connection using binary framing. HTTP/3 (finalized 2022) went further, replacing TCP itself with QUIC, a UDP-based transport designed to avoid head-of-line blocking.

Internal working: the request lifecycle

When you type a URL and hit enter, roughly this happens:

  1. DNS resolution — your OS asks a resolver to map the hostname to an IP address, often walking a chain of root, TLD, and authoritative name servers, cached at each level.
  2. TCP handshake — client and server exchange SYN, SYN-ACK, ACK to establish a reliable, ordered byte stream.
  3. TLS handshake — client and server negotiate a cipher suite and exchange keys (TLS 1.3 does this in one round trip instead of two) so everything after is encrypted.
  4. HTTP request/response — the client sends a method, path, and headers; the server (often Nginx, first) decides how to handle it.
ELI5 — The Restaurant

Looking up a domain name is like asking a directory service for a restaurant's address (DNS). Walking there and confirming you're both at the right table is the TCP handshake. Agreeing on a language and putting up a "private conversation" screen is TLS. Placing your order and getting food back is HTTP. Nginx is the maître d' who decides which kitchen station handles your order.

Level 1 — One sentence

The web works by layering naming, reliable transport, encryption, and a request/response protocol on top of each other.

Level 2 — One paragraph

A browser resolves a hostname via DNS, opens a reliable connection via TCP (or QUIC for HTTP/3), secures it via TLS, then sends an HTTP request describing what it wants. The server processes that request and returns a response with a status code, headers, and a body. Every layer is independently swappable — you can change TLS versions or HTTP versions without touching DNS.

Level 3 — Technical depth

Each layer has failure modes engineers must reason about: DNS caching and TTLs affect how fast changes propagate; TCP's congestion control affects throughput on lossy networks; TLS handshake cost affects latency for short-lived connections, which is why session resumption and TLS 1.3's 1-RTT matter; and HTTP/2's multiplexing removes application-layer head-of-line blocking but not TCP-layer blocking, which HTTP/3 (QUIC) finally fixes by making streams independent at the transport level.

Level 4 — Interview depth

Be ready to discuss: why HTTP/2 server push was deprecated in most browsers (poor cache alignment with actual client need), why QUIC uses UDP instead of extending TCP (deploying a new transport is easier in userspace over UDP than getting new TCP behavior adopted in kernels worldwide), and how connection reuse and keep-alive interact with load balancer algorithms upstream of Nginx.

curl -v https://example.com*   Trying 93.184.216.34:443...* Connected to example.com (93.184.216.34) port 443* TLS handshake, Client hello (1):* TLS handshake, Server hello (2):> GET / HTTP/1.1> Host: example.com< HTTP/1.1 200 OK
Common mistake

Assuming "HTTPS is slow" is inherent to encryption. In practice, the dominant cost is handshake round trips, not the cipher itself — which is why connection reuse, session resumption, and TLS 1.3 matter far more for perceived latency than CPU-bound encryption overhead.

Mnemonic

"Dumb Trucks Transport Heavy loads" — DNS, TCP, TLS, HTTP: the order every request travels through before an application ever sees it.

30-second revision

DNS resolves names → TCP/QUIC transports bytes → TLS encrypts them → HTTP gives them meaning. Nginx operates at the HTTP layer but its performance is shaped by everything beneath it.

5-minute revision

Walk through the full request lifecycle out loud: resolution, handshake, TLS negotiation, request, response. Name one thing that changed at each HTTP version (1.1: persistent connections, 2: multiplexing/binary framing, 3: QUIC/UDP transport). Explain why TLS 1.3 reduced handshake round trips.

What problem does HTTP/2 multiplexing solve?
Head-of-line blocking at the application layer — multiple requests share one TCP connection without waiting for each other in sequence.
Why does QUIC run over UDP instead of TCP?
TCP's in-order delivery guarantee means one lost packet blocks all streams on the connection. QUIC implements its own reliability per-stream over UDP, so a lost packet only blocks the stream it belongs to.

Interview & quiz

Explain what happens between typing a URL and seeing a page
DNS resolution → TCP handshake → TLS handshake → HTTP request sent → server processes and responds → browser parses HTML/CSS/JS and renders. Mentioning caching at each stage (DNS cache, TLS session resumption, HTTP cache headers) signals depth.
Is HTTP stateless? If so, how do sessions work? Trick
Yes, HTTP itself carries no memory between requests — state is bolted on via cookies, tokens, or server-side session stores keyed by an identifier the client resends. The trick is candidates often say "HTTP has sessions" when really the application layer simulates them.
Why might reducing TLS handshakes matter more than reducing payload size?
For small, frequent requests, round-trip latency from repeated handshakes can dominate total time far more than a few extra kilobytes of payload — this is why keep-alive and session resumption are high-leverage optimizations.

Meet Nginx & the C10K Problem

Fig 2.1 — Thread-per-connection vs. event-driven concurrency

Learning objectives

  • State what the C10K problem was and why it mattered
  • Explain why Nginx's architecture answered it differently than Apache
  • List Nginx's core roles: web server, reverse proxy, load balancer, TLS terminator, cache
  • Distinguish forward proxies from reverse proxies

Definition

Nginx (pronounced "engine-x") is an event-driven web server, reverse proxy, and load balancer. Rather than dedicating an OS thread or process to each connection, a small number of worker processes handle many thousands of connections concurrently using non-blocking I/O.

History: Igor Sysoev and the C10K problem

By the early 2000s, popular servers like Apache handled each connection with a dedicated process or thread. That model works fine at hundreds of concurrent connections, but as the web grew, engineers ran into what Dan Kegel named the C10K problem: how does a single server handle ten thousand concurrent connections? Thread-per-connection designs hit hard walls — context-switching overhead and per-thread memory (stacks, kernel structures) made scaling past a few thousand connections expensive or impossible on commodity hardware.

Igor Sysoev, a Russian systems engineer working at Rambler (a large Russian portal), began writing Nginx in 2002 specifically to solve this, releasing it publicly in 2004. Instead of one thread per connection, Nginx used the operating system's asynchronous event notification mechanisms directly, letting one process handle enormous numbers of connections with a small, fixed memory footprint.

Nginx's core roles

RoleWhat it means
Web serverServes static files (HTML, images, CSS) directly from disk very efficiently
Reverse proxySits in front of application servers, forwarding client requests to them
Load balancerDistributes requests across multiple backend servers
TLS terminatorHandles HTTPS encryption/decryption so backend apps don't have to
CacheStores responses to avoid repeating expensive backend work

Forward proxy vs. reverse proxy

Fig 2.2 — A forward proxy protects the client's identity; a reverse proxy protects the backend's
ELI5 — The Hospital

Imagine a hospital where every patient got assigned a dedicated nurse who does nothing else all day, even while the patient is asleep — that's thread-per-connection. Nginx works more like a triage desk: a handful of staff efficiently check on many patients in rotation, only spending active attention where it's needed right now. That's the event loop.

Level 1

Nginx handles massive numbers of connections with few processes by using non-blocking I/O instead of one thread per connection.

Level 2

Built by Igor Sysoev starting in 2002 to solve the C10K problem, Nginx uses an event-driven architecture, which lets it serve as a web server, reverse proxy, load balancer, TLS terminator, and cache — all while using far less memory per connection than thread-based servers like classic Apache.

Level 3

The event-driven model shines particularly for I/O-bound workloads (waiting on network or disk) since a worker isn't blocked while waiting — it services other connections' events in the meantime. CPU-bound work is a different story, which is why Nginx keeps its worker count close to the CPU core count rather than scaling workers with connection count.

Level 4

Be ready to explain that modern Apache (via mpm_event) narrowed this gap significantly, so the thread-vs-event distinction is more historical than a strict "Nginx always wins" claim — the real differentiator today is often Nginx's proxy/load-balancing feature set and configuration model, not raw connection handling alone.

Common mistake

Saying "Nginx is faster than Apache" as a blanket statement. It's more accurate to say Nginx's architecture handles high concurrency with lower memory overhead — for CPU-bound or module-heavy workloads the comparison is more nuanced.

Mnemonic

"WRLTC" — Web server, Reverse proxy, Load balancer, TLS terminator, Cache: the five hats Nginx wears.

30-second revision

C10K = the challenge of 10,000 concurrent connections. Nginx (Sysoev, 2004) solved it with an event-driven, non-blocking architecture instead of thread-per-connection.

5-minute revision

Explain the memory/context-switch cost of thread-per-connection at scale. Name Nginx's five core roles. Draw the forward vs. reverse proxy distinction from memory.

What is the C10K problem?
The challenge of a single server handling ten thousand concurrent connections efficiently, named by Dan Kegel, which thread-per-connection architectures struggled with.
Who created Nginx and why?
Igor Sysoev, starting in 2002 while working at Rambler, specifically to solve the C10K problem with an event-driven architecture.

Interview & quiz

What's the difference between a forward and reverse proxy?
A forward proxy sits in front of clients and hides their identity from servers (e.g., corporate proxies, VPNs). A reverse proxy sits in front of servers and hides backend topology from clients — Nginx is typically deployed as the latter.
Would Nginx's architecture help a CPU-bound video transcoding service? Trick
Not inherently — event-driven concurrency helps I/O-bound waiting, not CPU-bound computation. A transcoding service is usually better served by process pools sized to CPU cores, which Nginx itself does for its own worker count.

Master/Worker & the Event Loop

Fig 3.1 — One master, N worker processes, each running its own event loop

Learning objectives

  • Describe the master process's responsibilities vs. workers'
  • Explain epoll/kqueue and why they beat naive polling
  • Trace one iteration of the event loop for a real request
  • Explain graceful reloads and zero-downtime binary upgrades

Definition

Nginx runs as one privileged master process plus a configurable number of unprivileged worker processes. The master reads configuration, binds listening sockets, and manages workers; the workers do all the actual request handling, each running an independent event loop.

Why this split exists

Separating privilege from work is a security and stability pattern: the master can run as root (needed to bind port 80/443) while workers run as an unprivileged user, so a compromised worker doesn't automatically compromise the whole host. It also enables zero-downtime reloads — the master can start new workers with new configuration while old workers finish existing connections, then exit.

Internal working: the event loop

Each worker registers all its open sockets with the kernel's event notification interface — epoll on Linux, kqueue on BSD/macOS — instead of checking each socket in a loop (the old, slow select/poll approach that scales linearly with connection count). The worker then blocks on a single call (epoll_wait) that returns only the sockets that actually have activity. This is what makes handling thousands of idle-but-open connections nearly free.

A simplified loop, one iteration:

  1. Call epoll_wait() — sleep until the kernel reports ready file descriptors
  2. For each ready descriptor, run its associated state machine (e.g., "read more of the HTTP header," "write buffered response bytes")
  3. If a step would block (e.g., disk read not ready), register interest and move to the next ready descriptor rather than waiting
  4. Repeat

This is why Nginx modules must avoid blocking calls — a single blocking operation stalls every connection that worker is handling.

Master process responsibilities

TaskDetail
Read & validate configParses nginx.conf, exits with an error before disrupting running workers if invalid
Bind socketsOpens listening ports, which requires root for ports below 1024
Spawn/monitor workersForks worker processes and restarts them if they crash
Handle signalse.g. SIGHUP for graceful reload, SIGQUIT for graceful shutdown
Cache loader/managerSpawns short-lived helper processes that load and prune the on-disk cache

Graceful reload

Running nginx -s reload sends SIGHUP to the master. The master re-reads configuration, and if valid, starts a fresh set of workers using the new config while telling old workers to finish their current connections and exit — no dropped connections, no listening-socket downtime.

# Reload without downtimesudo nginx -t          # test config syntax firstsudo nginx -s reload   # sends SIGHUP to master
ELI5 — The Bank

The master process is the bank manager: opens the doors (binds sockets), hires tellers (workers), and can bring in a fresh shift without ever closing the branch (graceful reload). Each teller (worker) doesn't wait around staring at one customer — they glance at a board showing which windows have a customer ready (epoll), and serve whoever's ready next.

Level 1

One master manages configuration and processes; many workers each run an efficient event loop to handle connections.

Level 2

The master binds sockets and spawns workers, which use epoll/kqueue to be notified only about sockets with actual activity, avoiding the cost of checking every connection on every loop iteration. This lets a handful of workers handle tens of thousands of connections.

Level 3

Workers are single-threaded by default (though thread pools can be enabled for specific blocking operations like file I/O), and each runs a state machine per connection so no single client can occupy a worker exclusively. The worker_connections directive caps how many simultaneous connections one worker will track.

Level 4

Discuss the thread pool mechanism added in Nginx 1.7.11+, which offloads genuinely blocking operations (e.g., reading a file from slow disk) to a separate thread pool so it doesn't stall the whole event loop — a nuance that shows you understand event loops aren't magically immune to blocking I/O, just resistant to it by design.

Common mistake

Setting worker_processes far higher than the CPU core count "to handle more traffic." Since workers are largely I/O-bound and non-blocking, more workers than cores usually just adds context-switching overhead without adding throughput — auto (matching core count) is the standard recommendation.

Mnemonic

"One Manager, Many Tellers, One Ledger Each" — one master, many workers, each running its own independent event loop (ledger).

30-second revision

Master binds sockets and manages workers; workers run non-blocking event loops via epoll/kqueue. Graceful reload starts new workers with new config while old ones drain.

5-minute revision

Walk through one epoll_wait iteration end to end. Explain why blocking calls are dangerous inside a worker. Explain what SIGHUP does and why it enables zero-downtime reloads.

Why does Nginx use epoll instead of select/poll?
select/poll require scanning every registered socket each iteration (O(n)); epoll lets the kernel notify only about sockets with actual activity, which scales far better with high connection counts.
What happens to in-flight connections during a reload?
Old workers keep serving their existing connections until they naturally complete, while new workers (with new config) handle new connections — nothing is dropped.

Interview & quiz

Why does a single slow blocking call hurt Nginx more than it would hurt a thread-per-connection server? Trick
Because one worker's event loop is shared across potentially thousands of connections — blocking it stalls all of them, whereas in a thread-per-connection model a slow call only blocks that one thread/connection. This is exactly why Nginx offloads genuinely blocking work to thread pools rather than doing it inline.
What signal triggers a graceful reload, and what's the difference from a full restart?
SIGHUP triggers a graceful reload — new workers spin up with new config while old ones drain, with no dropped connections or socket downtime. A full restart tears down the master entirely, briefly closing listening sockets.

Configuration & Location Matching

Fig 4.1 — The context tree: http → server → location

Learning objectives

  • Explain Nginx's nested context model: http, server, location
  • Rank all location matching types by priority and give an example of each
  • Explain directive inheritance and where it breaks (e.g. array-like directives)
  • Read and reason about a full working nginx.conf

Definition

Nginx configuration is a nested, declarative tree of contexts. The outermost is the main context (global settings), inside it sits http (HTTP-wide settings), inside that one or more server blocks (virtual hosts), and inside those, location blocks that match specific request paths.

Why it's structured this way

Real deployments serve many domains and many distinct URL patterns from one machine — static assets, an API, a health check, a websocket endpoint — each needing different handling. Nesting lets shared settings live once at a high level (e.g., gzip settings in http) while specific overrides live close to what they affect (e.g., caching rules only for /images/).

Location matching — priority order

This is one of the most interview-tested details in all of Nginx. When multiple location blocks could match a URL, Nginx doesn't use the order they appear in the file — it follows a strict priority:

PrioritySyntaxMeaning
1 (highest)location = /pathExact match — stops searching immediately if matched
2location ^~ /pathPreferential prefix — if it's the longest matching prefix, regex checks are skipped
3location ~ /patternCase-sensitive regex, checked in file order, first match wins
3location ~* /patternCase-insensitive regex, same rules as above
4 (lowest)location /pathPlain prefix match — longest matching prefix wins if no regex matched
location = /health {          # 1. exact match, checked first    return 200 "ok";}location ^~ /static/ {        # 2. skips regex checks if this prefix matches    root /var/www;}location ~* \.(css|js)$ {     # 3. case-insensitive regex    expires 7d;}location /api/ {              # 4. plain prefix, lowest priority    proxy_pass http://backend;}
ELI5 — The Airport

Think of location matching like airport signage priority. A gate agent calling your exact name (=) beats a general boarding-zone sign (^~), which beats a sign matching a pattern like "all Zone B tickets" (regex), which beats a generic "this way to gates" arrow (plain prefix). Nginx always follows the most specific instruction available, not just the first one it sees.

Level 1

Nginx config nests http → server → location, and location blocks are matched by specificity, not file order.

Level 2

Exact matches (=) win outright; preferential prefixes (^~) skip regex evaluation if they're the longest matching prefix; regex matches are evaluated in file order; plain prefixes fall back to whichever is longest.

Level 3

Directive inheritance flows downward — a directive set in http applies to every server and location below unless overridden — but some directives (like add_header in older Nginx versions) don't merge additively; setting one in a child context can silently drop ones set in the parent, a classic gotcha.

Level 4

Be ready to trace a concrete ambiguous case live: given three overlapping location blocks, state out loud which one wins and why, citing the exact rule — this is one of the highest-frequency live-coding-style interview questions for Nginx roles.

Common mistake

Assuming location blocks are matched top-to-bottom like a switch statement. They're matched by specificity rules above — a regex location later in the file can override a prefix location declared earlier.

Mnemonic

"Exactly Preferred Regex Prefix" — Exact match, then ^~ Preferred prefix, then Regex, then plain Prefix: the location priority order, highest to lowest.

30-second revision

Config nests http → server → location. Location match priority: exact (=) > preferential prefix (^~) > regex (~ / ~*, file order) > longest plain prefix.

5-minute revision

Write three overlapping location blocks from memory and state which wins for a given URL. Explain one directive-inheritance gotcha (e.g. add_header not merging).

Which wins: location ^~ /images/ or location ~* \.png$ ?
location ^~ /images/ wins if it's the longest matching prefix — the caret-tilde modifier causes Nginx to skip regex evaluation entirely once it matches.
Are location blocks matched in the order they're written?
No — only regex locations of equal specificity are compared in file order; overall matching is governed by the exact > preferential-prefix > regex > prefix priority hierarchy.

Interview & quiz

Given `location /` and `location /images/`, which handles a request for `/images/logo.png`? Trick
`location /images/` wins because it's the longer matching prefix — plain prefix matches are resolved by longest match, not declaration order.
Why can `add_header` in a location block silently remove one set in the parent server block?
In many Nginx versions, `add_header` doesn't merge across nesting levels — if a child context defines its own `add_header`, it can replace rather than append to the parent's, unless the child re-declares everything needed. It's a well-known configuration trap.

Static Files & Caching

Fig 5.1 — The cache hierarchy, from browser to origin

Learning objectives

  • Explain sendfile() and why it avoids unnecessary copying
  • Configure gzip compression correctly, including common pitfalls
  • Distinguish Cache-Control, ETag, and Last-Modified and when each matters
  • Set up Nginx's own proxy cache for upstream responses

Definition

Static file serving is Nginx's original core competency: reading a file from disk and streaming it to a client as efficiently as possible, ideally without unnecessary memory copies, plus a layered set of caching mechanisms — browser-side, Nginx-side, and origin-side — that avoid redoing work.

sendfile() and why it matters

Without sendfile, serving a file naively means: read the file into userspace memory, then write that memory back out to the socket — two copies and two context switches between kernel and userspace. The sendfile() system call lets the kernel copy data directly from a file descriptor to a socket, skipping the userspace round-trip. Nginx enables this by default for static files.

server {    sendfile on;    tcp_nopush on;      # send headers and body together in one packet    gzip on;    gzip_types text/css application/javascript application/json;    gzip_min_length 1024;}

Gzip: what to compress and what not to

Compressing already-compressed formats (JPEG, MP4, most binary formats) wastes CPU for negligible size reduction. Gzip earns its keep on text formats: HTML, CSS, JavaScript, JSON, SVG. gzip_min_length avoids compressing tiny files where the compression overhead exceeds the savings.

Browser caching: three mechanisms

HeaderBehavior
Cache-Control: max-age=NBrowser reuses the cached copy for N seconds with zero network requests
ETagA fingerprint of content; browser sends it back via If-None-Match, server replies 304 if unchanged, avoiding a full re-download
Last-ModifiedA timestamp-based, coarser-grained cousin of ETag, used the same way via If-Modified-Since

Nginx as a cache: proxy_cache

Beyond serving its own static files, Nginx can cache responses from an upstream application, so repeated identical requests never reach the backend at all.

proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=my_cache:10m max_size=1g;server {    location /api/ {        proxy_cache my_cache;        proxy_cache_valid 200 10m;        proxy_pass http://backend;    }}
ELI5 — The Library

Your browser's cache is like keeping a photocopy of a book at home — no trip needed at all. Nginx's cache is the local library branch keeping a copy so you don't have to go to the central archive across town. sendfile is the librarian handing you the book directly off the shelf instead of first photocopying it in the back room, then handing you the copy.

Level 1

Serve static files efficiently with sendfile, compress text with gzip, and let cache headers or Nginx's own proxy cache avoid repeating work.

Level 2

sendfile avoids userspace copies for file serving; gzip trades CPU for bandwidth on compressible text formats; Cache-Control/ETag/Last-Modified let browsers skip re-fetching unchanged content; proxy_cache lets Nginx itself absorb repeated upstream requests.

Level 3

ETag validation still requires a round trip (a conditional GET), it just avoids re-transferring the body — that's different from max-age, which avoids the network round trip entirely. Choosing between them is a tradeoff between freshness guarantees and request volume.

Level 4

Discuss cache stampede/thundering herd risk when a popular cached object expires — proxy_cache_lock in Nginx serializes concurrent regeneration requests to the same key so only one request repopulates the cache while others wait, rather than all hammering the backend simultaneously.

Common mistake

Enabling gzip for all MIME types indiscriminately, including images and video, which burns CPU compressing data that's already compressed and gains almost nothing in size.

Mnemonic

"Send it, Zip it, Tag it, Cache it" — sendfile, gzip, ETag, proxy_cache: the four static-serving levers in order of where they act.

30-second revision

sendfile avoids userspace copies; gzip compresses text formats only; ETag/Last-Modified enable conditional requests; proxy_cache lets Nginx cache upstream responses directly.

5-minute revision

Explain the difference between max-age and ETag-based caching in terms of network round trips. Explain what proxy_cache_lock prevents and why.

Why shouldn't you gzip JPEG images?
JPEG is already a compressed format — running gzip on it burns CPU for negligible or even negative size savings.
What does proxy_cache_lock prevent?
A thundering herd / cache stampede — when a cached entry expires, it ensures only one request regenerates it while concurrent requests for the same key wait, rather than all hitting the backend simultaneously.

Interview & quiz

What's the practical difference between Cache-Control: max-age and ETag? Trick
max-age avoids a network request entirely until it expires; ETag still requires a round trip (a conditional GET) but avoids re-downloading the body if unchanged. They're often used together — max-age for the common case, ETag as a fallback validator once max-age expires.
Why might sendfile need to be disabled in some proxy configurations?
sendfile serves files directly from disk to socket — it's not relevant when Nginx is proxying to an upstream rather than serving a local file, and in some SSL/streaming edge cases it can be disabled to allow other processing on the data in transit.

Reverse Proxy & Load Balancing

Fig 6.1 — Nginx distributing requests across a backend pool

Learning objectives

  • Configure proxy_pass and know which headers must be forwarded manually
  • Compare load balancing algorithms: round robin, least connections, ip_hash, weighted
  • Explain health checks (passive vs. active) and failover behavior
  • Understand session persistence and when it's actually needed

Definition

As a reverse proxy, Nginx accepts client requests and forwards them to one or more backend servers (the "upstream"), returning the backend's response to the client as if Nginx had produced it itself. Load balancing is the policy Nginx uses to pick which backend gets each request when there's more than one.

The basics: proxy_pass and forwarded headers

By default, the backend sees the connection as coming from Nginx, not the original client — so Nginx must explicitly forward useful context via headers.

upstream backend {    server 10.0.0.1:8080;    server 10.0.0.2:8080;}server {    location / {        proxy_pass http://backend;        proxy_set_header Host $host;        proxy_set_header X-Real-IP $remote_addr;        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;        proxy_set_header X-Forwarded-Proto $scheme;    }}

Load balancing algorithms

AlgorithmDirectiveBest for
Round robin(default)Uniform backends, roughly equal request cost
Least connectionsleast_conn;Requests with variable processing time
IP haship_hash;Simple session persistence without shared session storage
Weightedserver ... weight=3;Heterogeneous backend capacity

Health checks and failover

Open-source Nginx does passive health checks by default: it marks a backend as unavailable after a configured number of failed requests within a time window, then periodically retries it.

upstream backend {    server 10.0.0.1:8080 max_fails=3 fail_timeout=30s;    server 10.0.0.2:8080 max_fails=3 fail_timeout=30s;    server 10.0.0.3:8080 backup;   # only used if all primaries are down}

Active health checks — where Nginx proactively probes a health endpoint on a timer regardless of live traffic — require Nginx Plus or third-party modules; this is a real, common gap that trips people up when they assume open-source Nginx does this out of the box.

ELI5 — The Bank, Revisited

Load balancing is the "next available teller" rope-line system. Round robin cycles through tellers in order; least-connections sends you to whichever teller currently has the shortest line; ip_hash always sends the same customer to the same teller (useful if that teller kept notes about you that no one else can see).

Level 1

Nginx forwards requests to backend servers and picks among them using an algorithm like round robin, least connections, or ip_hash.

Level 2

proxy_pass forwards the request; proxy_set_header preserves client context the backend would otherwise lose; load balancing algorithms trade off simplicity, fairness under variable load, and session affinity needs.

Level 3

ip_hash-based session persistence breaks down behind another layer of NAT or a corporate proxy where many clients share one IP — all of them get pinned to the same backend, defeating the point of load balancing. Cookie-based session persistence (via Nginx Plus or app-level solutions) avoids this.

Level 4

Be ready to discuss why "stateless backends behind a stateless load balancer" is the preferred production pattern over session persistence altogether — pushing session state to a shared store (Redis, a database) removes the load balancer's need to care about affinity at all, which is more robust and lets you freely add/remove backend capacity.

Common mistake

Forgetting proxy_set_header X-Forwarded-For, then being surprised when application logs show every request coming from Nginx's own IP address instead of real client IPs.

Mnemonic

"Round, Least, Hash, Weight" — the four load balancing algorithms in the order most engineers learn them.

30-second revision

proxy_pass forwards requests; proxy_set_header preserves client context; load balancing algorithms are round robin, least_conn, ip_hash, and weighted. Passive health checks are built in; active checks need Nginx Plus.

5-minute revision

Write an upstream block with three servers and a backup. Explain why ip_hash can create uneven load behind shared-NAT clients. Explain why stateless backends are the preferred pattern over session persistence.

What header tells the backend the client's real IP?
X-Forwarded-For (and often X-Real-IP), set explicitly via proxy_set_header since the backend otherwise only sees Nginx's IP.
Does open-source Nginx support active health checks out of the box?
No — only passive health checks (based on real request failures) are built into open-source Nginx; active probing requires Nginx Plus or third-party modules.

Interview & quiz

Why might ip_hash produce uneven load in practice? Trick
If many clients share a single public IP (corporate NAT, mobile carrier NAT), they all hash to the same backend, defeating the balancing goal — a common gotcha interviewers probe for.
What's the production-preferred alternative to session persistence at the load balancer?
Making backends stateless and storing session data in a shared external store (e.g. Redis), so any backend can serve any request — removing the load balancer's need for affinity entirely.

TLS & Security

Fig 7.1 — TLS termination: encryption ends at Nginx, internal traffic can be plain HTTP on a trusted network

Learning objectives

  • Configure TLS termination and understand certificate chain basics
  • Explain rate limiting with limit_req and limit_conn
  • Harden headers against common attacks (clickjacking, MIME sniffing)
  • Understand why re-encrypting to the backend is sometimes still necessary

TLS termination

Nginx commonly acts as the TLS termination point: it holds the certificate, handles the handshake, decrypts incoming traffic, and forwards plain HTTP internally over a trusted private network — sparing every backend instance from managing certificates itself.

server {    listen 443 ssl;    server_name example.com;    ssl_certificate     /etc/ssl/certs/example.com.pem;    ssl_certificate_key /etc/ssl/private/example.com.key;    ssl_protocols TLSv1.2 TLSv1.3;    ssl_prefer_server_ciphers on;}

In regulated or zero-trust environments, traffic may be re-encrypted from Nginx to the backend too (TLS re-encryption, sometimes called "double SSL"), trading some performance for defense-in-depth against anyone who gains access to the internal network.

Rate limiting

Nginx can throttle abusive or runaway clients using a shared-memory zone that tracks request rate per key (usually client IP).

http {    limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;    server {        location /api/ {            limit_req zone=api_limit burst=20 nodelay;        }    }}
Common mistake — worth memorizing

limit_req_zone must be declared in the http block — it defines a shared memory zone, which is a global resource. Declaring it inside a server block is invalid and will fail config validation; only the usage directive, limit_req, belongs inside server/location.

Security headers

HeaderDefends against
X-Frame-Options: DENYClickjacking via iframe embedding
X-Content-Type-Options: nosniffMIME-sniffing based attacks
Strict-Transport-SecurityDowngrade/SSL-stripping attacks by forcing HTTPS on future visits
Content-Security-PolicyCross-site scripting by restricting allowed content sources
ELI5 — The Mall Security Desk

TLS termination is like a mall having one secure entrance with a metal detector (Nginx) instead of requiring every single store inside to run its own airport-style screening. Rate limiting is the fire-code capacity limit on how many people can enter through that door per minute.

Level 1

Nginx terminates TLS so backends don't each need certificates, and can throttle abusive traffic with rate limiting.

Level 2

TLS termination centralizes certificate management and handshake cost at the edge; rate limiting via a shared memory zone protects backends from being overwhelmed; security headers close common browser-side attack vectors cheaply.

Level 3

The burst and nodelay parameters on limit_req matter: without nodelay, burst requests are queued and delayed rather than rejected, which can be the wrong choice for latency-sensitive APIs versus a strict reject-over-quota approach.

Level 4

Discuss OCSP stapling (letting Nginx include certificate revocation status in the handshake itself, saving the client an extra round trip to the CA) and mutual TLS (mTLS) for service-to-service authentication in zero-trust architectures — both are realistic senior-level follow-up questions.

Mnemonic

"Zone in the House, Limit at the Door" — limit_req_zone belongs in http (the house/global scope); limit_req is applied at the door (server/location).

30-second revision

Nginx terminates TLS at the edge, optionally re-encrypting to backends in zero-trust setups. limit_req_zone (in http) + limit_req (in server/location) rate-limit clients. Security headers close common browser attack vectors.

5-minute revision

Write a valid limit_req_zone + limit_req pair from memory, correctly scoped. List three security headers and what each defends against. Explain what nodelay changes about rate limiting behavior.

Where must limit_req_zone be declared?
In the http block — it's a global shared-memory-zone declaration, not a per-server or per-location directive.
What does Strict-Transport-Security (HSTS) prevent?
Downgrade and SSL-stripping attacks, by telling the browser to only ever connect via HTTPS for a specified duration, even if a user types or clicks an http:// link.

Interview & quiz

Spot the bug: `limit_req_zone` is placed inside a `server` block. What happens? Trick
Config validation fails — `limit_req_zone` defines a shared memory zone and must live in the `http` context; only `limit_req` (which references the zone by name) is valid inside `server`/`location`.
Why would a zero-trust environment still encrypt traffic between Nginx and backend servers on the same private network?
Defense-in-depth — if an attacker gains a foothold inside the internal network, unencrypted internal traffic becomes trivially interceptable; re-encrypting assumes the internal network itself isn't fully trusted.

Performance Tuning & the Full HA Reference Architecture

Fig 8.1 — A reference HA architecture: active/standby load balancers, a backend pool, shared state

Learning objectives

  • Tune worker_processes, worker_connections, and keepalive settings deliberately
  • Identify and fix the most common real-world Nginx bottlenecks
  • Design a highly available Nginx layer, not just a highly available backend
  • Synthesize every prior chapter into one coherent production architecture

Core tuning knobs

DirectiveWhat it controlsGuidance
worker_processesNumber of worker processesauto — match CPU core count
worker_connectionsMax simultaneous connections per workerTune with expected concurrency; watch ulimit -n
keepalive_timeoutHow long idle client connections stay openBalance connection reuse against tying up workers
keepalive (upstream)Idle connections kept open to backendsReduces repeated TCP/TLS handshake cost to upstreams
worker_processes auto;events {    worker_connections 4096;    use epoll;}upstream backend {    server 10.0.0.1:8080;    server 10.0.0.2:8080;    keepalive 32;   # reuse connections to backends}

Common real-world bottlenecks

  1. File descriptor limits — the OS-level ulimit capping connections well below worker_connections
  2. No upstream keepalive — re-handshaking TCP/TLS to backends on every request
  3. Oversized worker_processes — more workers than cores, adding context-switch overhead with no throughput gain
  4. Buffering misconfiguration — proxy buffers too small for response sizes, causing excessive disk temp-file writes

The full reference architecture

Bringing every chapter together: DNS or a cloud load balancer distributes traffic across an active/standby pair of Nginx instances (using something like keepalived and VRRP to share a virtual IP, so a failed primary doesn't take the whole edge down). Each Nginx instance terminates TLS, applies rate limiting and security headers, and load-balances across a pool of stateless application servers using least-connections or round robin. A shared cache/session store (Redis or similar) means any backend can serve any request, and Nginx's own proxy_cache absorbs repeated read-heavy traffic before it ever reaches that backend pool.

ELI5 — Putting It All Together

The whole system is a relay: a directory service points you to the right building (DNS), a backup security desk exists in case the main one calls in sick (active/standby LB), the front desk checks IDs once so nobody inside has to (TLS termination), a "next available" queue distributes visitors fairly (load balancing), and a shared notice board means any staff member can help any visitor without needing personal history (shared session store).

Level 1

Tune worker counts and connection limits to match hardware, then make the whole Nginx layer itself redundant, not just the backend.

Level 2

worker_processes should generally match core count; worker_connections should be sized against realistic concurrency and OS file descriptor limits; keepalive to upstreams avoids repeated handshake costs; and a single Nginx instance is a single point of failure unless paired with a failover mechanism like keepalived/VRRP.

Level 3

Explain why the architecture pushes state out of both the load balancer (no session persistence needed) and increasingly out of the application servers themselves (shared external session store) — this is what allows horizontal scaling of the backend pool without touching the Nginx layer's configuration.

Level 4

Be ready to discuss trade-offs at the edge: active/standby (simpler, wastes standby capacity) versus active/active with anycast or ECMP (better utilization, more operational complexity) — a senior-level question tests whether you can articulate why a design decision was made, not just recite the final diagram.

Common mistake

Treating "we have multiple app servers" as sufficient high availability while running a single Nginx instance in front of them — the load balancer itself is now the single point of failure the architecture forgot to remove.

Mnemonic

"Balance the Balancer" — high availability isn't done until the load balancer itself has a failover partner, not just the backends behind it.

30-second revision

Tune worker_processes to core count, size worker_connections against real concurrency and OS limits, keep connections alive to upstreams, and make the Nginx layer itself redundant via active/standby with a shared virtual IP.

5-minute revision

Draw the full reference architecture from memory: DNS/LB entry → active/standby Nginx pair → TLS termination + rate limiting → load-balanced stateless app pool → shared cache/session store. Explain what single point of failure it removes at each layer.

Why pair Nginx with keepalived in a production edge design?
A single Nginx instance is a single point of failure; keepalived (via VRRP) lets a standby instance take over a shared virtual IP if the primary fails, removing that SPOF.
What's the throughput cost of not using upstream keepalive?
Every request to a backend re-does a full TCP (and possibly TLS) handshake instead of reusing an existing connection, adding latency and CPU overhead that compounds under high request rates.

Interview & quiz

Your app servers are horizontally scaled and healthy, but the site still goes down during a traffic spike. What's a likely architectural gap? Trick
A single, non-redundant Nginx/load-balancer instance in front of the scaled app tier — the backend scaling was solved but the edge itself remains a single point of failure, which is exactly the trap this chapter targets.
Why does moving session state out of the application server matter for horizontal scaling, not just for the load balancer?
If session state lives on a specific app server, that server becomes a soft dependency for that user's future requests even without explicit load-balancer affinity — externalizing state to a shared store makes every app server instance truly interchangeable, which is what horizontal scaling actually requires.
